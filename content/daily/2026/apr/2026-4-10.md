### StablePay
云效流水线在构建镜像时，拉取 golang:1.26.1-alpine 和 alpine:latest 时网络超时。这是因为云效构建环境（通常位于阿里云内部）访问 Docker Hub 不稳定。
```bash
docker pull golang:1.26.1-alpine
docker pull alpine:latest

docker tag golang:1.26.1-alpine stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/golang:1.26.1-alpine
docker tag alpine:latest stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/alpine:latest

docker push stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/golang:1.26.1-alpine
docker push stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/alpine:latest
```
然后修改6个微服务的Dockerfile：
```bash
# 原来
FROM golang:1.26.1-alpine AS builder
...
FROM alpine:latest

# 改为
FROM stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/golang:1.26.1-alpine AS builder
...
FROM stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/alpine:latest
```

### 本科毕设

```
这次不要再发散新 idea，也不要重写问题定义。你现在只做两件事：

1. 搭建 A5：在 A4 的 gateway 子集上，补做“第二阶段 follow-up replay”
2. 评估是否需要重跑 A4，以及如果重跑，样本量从32扩到多少最合适 

先说我的明确要求：

- 不改 num_history
- 不改训练
- 不改 single-vLLM 主路径
- 不引入 student / distillation / token budget
- 不做新的大重构
- 继续沿用 A4 的 gateway target 和三种弱干预
- 所有新逻辑必须挂在显式开关后，默认关闭
- 先做最小可运行版本，再谈分析

================================
任务一：实现 A5 = two-stage follow-up replay
================================

你现在已经有 A4 的 gateway 子集：
- 第一次 call：带 history
- baseline 输出是单个 ↓
- 同 step_id 后续 follow-up call 会输出 pixel_goal

A5 要做的是：
对每个 selected gateway event、每个 history_index、每个 intervention_variant，
在 probe 完第一阶段 gateway call 后，
如果 probe 后仍然保持为“单个 ↓”，
就继续 replay 第二阶段 follow-up call，
然后看最终 pixel_goal 是否还能保住，以及坐标偏移多大。

------------------------
A5 的最小实现要求
------------------------

【1】新增显式开关
例如：
- history_probe_run_followup_replay = True / False
默认 False，A5 配置里设 True

【2】第一阶段仍然沿用 A4
也就是：
- target 仍然是 history_conditioned_gateway_only
```
