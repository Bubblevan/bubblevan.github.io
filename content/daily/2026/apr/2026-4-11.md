好问题！这是一个重要的观察。让我解释一下：

原因分析

这是 StablePay 后端 DID 服务的行为，不是 bug。当你调用 stablepay_register_local_did 时：

1. 你提交的本地钱包信息：
    - DID: did:solana:EBv2e3SjceE6dkxSs5qbbhtFVvsrux5dgmM2cAJw9RsF
    - 钱包地址: EBv2e3SjceE6dkxSs5qbbhtFVvsrux5dgmM2cAJw9RsF
2. 后端 DID 服务返回的：
    - Backend DID: did:solana:57jmMRqDzUyjaUZLf7kZBgUFzx9ngR3LxonC4gRuSPgg
    - 钱包地址: 57jmMRqDzUyjaUZLf7kZBgUFzx9ngR3LxonC4gRuSPgg

为什么 DID 会变化？

从插件代码和 StablePay 架构来看，可能的原因：

### 1. 后端重新生成了密钥对

后端 DID 服务可能没有在本地存储你的原始公钥，而是为你生成了新的密钥对。这 在 stablepay_register_local_did 的实现中可以看到：

    ```javascript
    registered = await client.registerLocalDid({
        user_type: params.user_type ?? "agent",
        public_key: status.wallet.wallet_address,  // 你提交的 EBv2...
        wallet_address: status.wallet.wallet_address,
        wallet_id: status.wallet.wallet_id,
        ...
    });
    ```

后端可能：
- 没有验证/使用你提交的 public_key
- 而是自己生成了新的密钥对（57jm...）
- 返回了新的 DID

### 2. DID 服务的设计问题

从返回的数据看：

    ```json
    {
        "did": "did:solana:57jmMRqDzUyjaUZLf7kZBgUFzx9ngR3LxonC4gRuSPgg",
        "public_key": "57jmMRqDzUyjaUZLf7kZBgUFzx9ngR3LxonC4gRuSPgg",
        "wallet_address": "57jmMRqDzUyjaUZLf7kZBgUFzx9ngR3LxonC4gRuSPgg"
    }
    ```

注意这三个值是相同的，这表明后端可能：
- 没有正确区分 DID、公钥和钱包地址的概念
- 或者使用了简化的实现，直接返回新生成的地址

### 3. 这可能是个 Bug

从插件的设计意图来看，backend_did 应该与你的本地钱包 DID 保持一致，或者至少是同一个密钥对的不同表示。但现在：
- 本地钱包 DID: EBv2...
- 后端 DID: 57jm...

这意味着你无法用本地钱包签名来验证后端 DID，因为它们的密钥对不同。

影响

## 一、和你现在观察最接近的先例

### 1) StreamVLN：已经明确把“近期活跃上下文”和“慢更新历史记忆”分开

StreamVLN 提出的是 **slow-fast context modeling**：
快速流用 active dialogue 的 sliding window 保持响应性，慢速流把历史视觉状态压缩进 slow-updating memory，还用 3D-aware token pruning 来减冗余。这个思路和你现在看到的
“recent 更像 gateway 的即刻支撑，older history 更像补充性背景”
在概念上是非常接近的。([arXiv][4])

### 2) JanusVLN：最像你“recent sufficiency + early complementarity”的结构先例

JanusVLN 直接把 memory 拆成 **spatial-geometric** 和 **visual-semantic** 两类固定大小隐式记忆，并且显式说只保留 **initial tokens + sliding window tokens** 的 KV 来做增量更新。这个 “**初始部分 + 最近窗口**” 的保留策略，和你现在看到的
“最早几帧不是单独最强，但在完整上下文里很补；最近几帧单独最能撑住”
是目前文献里最像的一类结构性先例。([OpenReview][5])

### 3) ProVideLLM：长时语义历史 vs 短时视觉上下文

ICCV 2025 的 Streaming VideoLLMs for Real-Time Procedural Video Understanding 明确把 streaming past observations 分成 **long-term semantic history** 和 **short-term visual context**，并指出长时历史会被 verbalize/compress 成语义序列，而短时部分保留细粒度视觉信息。更关键的是，它还说这种 long-term semantic compression **会增强当前 step prediction**。这几乎就是你“早期 history 提供补充约束、近期 history 提供即刻可用视觉支撑”的视频版先例。([CVF开放获取][1])

### 4) WeaveTime：明确提出 past-current focus blindness

WeaveTime 不是导航，但它直接点名一个和你问题高度同构的 failure：
**past-current focus blindness**，即模型无法区分当前观察与累积历史。它为此引入 Past-Current Dynamic Focus Cache，只在需要时扩展历史。这个工作至少说明：
“当前 vs 历史” 的角色混淆，本来就是 streaming MLLM 的已知问题；所以你现在看到 history 对 gateway 的分工，并不是空想。([arXiv][3])

### 5) LION-FS：response determination 和 detailed generation 的两阶段分工

LION-FS 虽然是在线视频助手，不是 VLN，但它把**快速 response determination** 和 **慢速、精细的 response generation** 明确拆开：Fast Path 负责快速决定，Slow Path 则在当前关键帧上做更细的多粒度增强。它甚至明确说 Slow Path 的 keyframe 是“当前决定响应的帧”。这和你现在 DualVLN 里
“先 gateway 决定是否 `↓`，再 follow-up 给更细 subgoal”
在功能分工上非常像。

## 二、支持“history 不是等价的”这件事的先例

### 6) RoboTron-Nav：adaptive 3D-aware history sampling

RoboTron-Nav 直接提出 **adaptive 3D-aware history sampling**，目标就是“有效且高效地利用历史观察”。这至少说明在导航里，历史观测本来就不该被当成等价输入。

### 7) Memory-Adaptive VLN：动态抑制有害 history，只保留有利内容

Memory-Adaptive Vision-and-Language Navigation（Pattern Recognition 2024）明确把问题写成：history 对决策很重要，但简单加权会把有害内容一起带进来；它的方法是动态抑制历史中的 harmful content，只保留对导航有利的信息。这个方向没有你现在这么细的 temporal-role 发现，但它支持“**早期 history 可能不是没用，而是混着有用/有害信息**”这种思路。

### 8) GOAT：history 里有 confounders，不能直接等价使用

CVPR 2024 的 GOAT（Vision-and-Language Navigation via Causal Learning）明确把 **vision / language / history** 里的 observable 和 unobservable confounders 都拿出来建模，用 back-door / front-door adjustment 去减轻 spurious correlations。它没有证明你的 “early complementary / recent sufficient”，但它从因果角度支持了一件事：
**history 里的内容不是天然可靠或等价的，里面会有真正干扰决策的成分。** ([CVF开放获取][8])

### 9) LH-VLN / MGDM：短期与长期 memory 的动态协同

CVPR 2025 的 Towards Long-Horizon VLN 提出 MGDM，明确写成 **dynamic long-term and short-term memories**。它和你的实验不在同一 granularity，但它属于“长程 VLN 里短期/长期记忆角色不同”的直接先例。([CVF开放获取][9])

## 三、支持“视频 token 的时间角色不一样”这件事的先例

这几篇不是导航论文，但对你现在这个现象的解释很有帮助。

### 10) FrameFusion：similarity + importance，不是简单时间近就行

FrameFusion 的核心不是“最近最重要”，而是要同时考虑 **similarity** 和 **importance**。它分析了视频 token 在层间的相似性分布，并用 similarity-based merging + importance-based pruning。对你来说，它支持的是：
**时间位置本身不是全部；某些较早帧虽然不最新，但可能在完整上下文里提供不可替代的信息。** ([CVF开放获取][10])

### 11) Dynamic-LLaVA：prefill 阶段的视觉上下文冗余可动态稀疏化

Dynamic-LLaVA 明确把多模态推理里的视觉上下文冗余当作 prefill 大头，并提出 dynamic vision-language context sparsification。它没告诉你哪一帧最关键，但它支持：
**history 不该按等预算全送进去，应该是动态预算。** ([arXiv][11])

### 12) PruneVid：temporal static + spatial similarity + query relevance

PruneVid 的做法更贴你的 taste：它一边 merge temporally static / spatially similar tokens，一边利用 query relevance 去保留和问题相关的视觉 token。对你现在的现象，一个很自然的文献支撑是：
**older frames 之所以有补充价值，可能不是因为“老”，而是因为它们承载了 query-relevant 但当前帧不再重复的内容。** ([ACL Anthology][12])

### 13) M-LLM Based Video Frame Selection：frame relevance 可以按 query 学出来

CVPR 2025 这篇直接做 adaptive frame selection，目标就是选和用户 query 更相关的帧。它不证明你的 gateway 机制，但它给了一个直接先例：
**“哪些 history 更重要”这件事，本来就可以被建模，而不是只能凭 recency 假设。** ([CVF开放获取][13])

## 四、支持你“做 probe 分析”这一方法本身的先例

### 14) On the Consistency of Video LLMs in Temporal Comprehension

这篇 CVPR 2025 对你很有帮助，不是因为它研究导航，而是因为它**专门做 probe**：
先让模型给出一个 temporal grounding，然后再通过一系列 probes 检查模型的回答是否与初始 grounding 一致。论文结论是当前 Video-LLMs 对视频内容、query 改写、任务设置都很敏感，而且 consistency 很差。
这给你两个很强的背书：

1. 对多模态/视频模型做这种**counterfactual consistency probe** 是正统方法；
2. 即使模型最终输出看起来对，内部 temporal decision boundary 也可能很不稳。([CVF开放获取][14])

### 已经被提出过的部分

### 但我没找到被明确证明过的部分

我**没有找到**哪篇公开论文已经明确做了下面这件事：

* 在 **VLN / DualVLN 风格的两阶段决策**里，
* 先锁定 **gateway call**（不是最终 pixel-goal call），
* 再通过**弱 history 干预**发现：

  * 最近 history 更能单独支撑 gateway，
  * 早期 history 在完整上下文里提供补充约束，
  * 并且这件事具体作用在 “是否进入 `lookdown -> pixel_goal` 通路” 上。

换句话说：

**你的 broad idea 不新；但你的细粒度现象和落点，目前看仍是新鲜的。**
