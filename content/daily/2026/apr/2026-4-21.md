#### Agent 金标准
这里记录一下，免得又给我Team账号封了：
**“Agent Harness” 现在很火，但它更像一个正在成形的工程概念，而不是像 Transformer、POMDP、RLHF 那样边界清晰的学术术语。** 我联网查下来，最近把这个词讲得最系统的是 Anthropic：他们在 2025 年底写了 *Effective harnesses for long-running agents*，2026 年 3 月又写了 *Harness design for long-running application development*，还在 2026 年 1 月把 **agent harness / scaffold** 明确定义为“让模型表现为 agent 的那套系统：处理输入、编排工具调用、返回结果”。换句话说，**模型不是 agent；模型 + harness 才是 agent 系统。** 

所以我会把 **Agent Harness** 翻成：**智能体脚手架 / 运行编排层 / 外骨骼系统**。它管的不是“模型会不会想”，而是“模型怎么跑起来”：上下文怎么装配、工具怎么暴露、任务怎么拆、状态怎么跨 session 交接、失败怎么恢复、谁来验证结果、怎么计费、怎么审计。Anthropic 甚至直接说，harness design 是前沿 agentic coding 性能的关键；他们后续又补了一句：harness 里其实编码了“模型现在还做不到什么”的假设，而这些假设会随着模型升级快速过时。
这也是为什么你会觉得那篇 *Harness design for long-running application development* “更多是经验，没啥实质性内容”——**它确实不是理论论文，而是一篇高密度的 systems note。** 但它并不空，因为它把几个非常核心的系统问题讲透了：长任务里模型会随着 context 变长而失去一致性；compaction 不等于 reset；要靠结构化 handoff artifact 维持跨 session 连续性；自评往往失真，所以把生成器和评估器拆开是强杠杆；再往上可以变成 planner / generator / evaluator 的三 agent 结构。这个“没啥理论味”的地方，恰恰就是现在 Agent 研究最有含金量的地方：**不是只问模型能不能答对，而是问系统怎么让它连续几小时不跑偏。** 

##### “Agent 开发”和“Agent 算法”

我建议你以后不要再用“会不会搭框架”来理解 Agent，不然确实会觉得“只是工程活”。更好的拆法是：

**Agent 开发**关心的是：
怎么把模型接进环境、工具、文件系统、浏览器、终端、数据库；怎么管理上下文、任务状态、权限、安全、日志、评测、部署和成本。Anthropic 对 harness、context engineering、tool design、eval harness 的一整串文章，本质上都在回答这些问题。

**Agent 算法 / 研究**关心的是：
怎么让 agent 在长时序、多步、多工具、部分可观测、稀疏奖励、环境可变的条件下，仍然能稳定规划、检索、记忆、反思、学习和协作。过去三年里，真正有学术味的部分主要集中在：**记忆机制、上下文管理、规划与 workflow 生成、agent-computer interface、benchmark/eval、RL/post-training、多 agent 协调、安全与可控性**。这几条在近两年的 benchmark 和训练论文里都非常明显。

所以，“Agent 研究是不是一直和可解释性粘在一起？”——**不是。** 可解释性只是很早期的一条副线，比如 ReAct 那类工作会强调 reasoning trace 更可读；但 2024–2026 的主线已经明显转向了 **可控性、可持续性、可评估性、可训练性、可部署性**。Anthropic 甚至把 eval harness、context engineering、tool contract 这些看起来“不学术”的东西都写成了核心方法论，因为现在真正难的是让 agent 在真实环境里稳定工作，而不是只在单轮 prompt 上表现好。

##### 最该建立的心智模型

我会把 Agent 系统先记成这 6 个块：

**1. Brain（模型）**
负责推理、生成、规划，但自己并不等于系统。Anthropic 明确把 harness 和 model 分开谈，评估 agent 时评的是两者一起。

**2. Context（上下文工程）**
现在已经不只是 prompt engineering 了。Anthropic 在 2025 年把 **context engineering** 定义成：围绕整个推理时刻，持续策划“哪些 token 应该进上下文”的策略，包括 system instructions、tool schema、MCP、外部数据、message history 等。长任务里他们重点提 compaction、structured note-taking、sub-agent architectures。

**3. Memory（记忆）**
从论文角度，MemGPT 把它类比成操作系统里的层级内存管理；HiAgent 则更进一步，专门研究**工作记忆**，把长任务中的 action-observation history 变成以 subgoal 为单位的层级 chunk。你可以把 “memory” 分成短期 working memory、跨试验/跨会话 memory、以及检索式 external memory 三层。

**4. Interface（工具与环境接口）**
SWE-agent 的核心贡献不是“又一个 coding agent”，而是提出 **ACI, agent-computer interface**：给 agent 什么命令集合、返回什么格式的环境反馈，会直接影响性能。这个洞见非常学术，因为它把“工具设计”从产品细节变成了可研究对象。

**5. Evaluation（评测）**
AgentBench、WebArena、VisualWebArena、OSWorld、AppWorld、GAIA、MLGym 这一串工作都在说明：没有 benchmark 和 grader，agent 研发会非常“玄学”。Anthropic 2026 年也专门把 eval harness 和 agent harness 区分开了。

**6. Learning（训练与优化）**
这部分是最容易被“只会搭框架”的人忽略的。现在已经出现了明显把 agent 训练问题单独拿出来做的工作，比如 Agent Lightning，它把 agent execution 和 RL training 解耦，试图给“任意 agent”做统一 RL 接口和 credit assignment。

##### 如果以后想发 Agent 方向论文，选题应该怎么想

我不建议你把自己困在“要么开发、要么算法”的二分里。Agent 最适合发论文的点，通常长在**系统问题被形式化之后**。我给你一个很实用的选题地图：

1. Memory / Context

该记什么？何时记？如何压缩？如何检索？何时 reset、何时 compact、何时写外部 note？MemGPT、HiAgent、Anthropic 的 context engineering 都在打这条线。

1. Planning / Workflow

workflow 是人写、模型写、还是系统自己搜索出来？怎么从 chain planning 扩展到 graph planning / branching workflow？WorFBench 和 AFlow 都很贴这条线。

1. Agent-Computer Interface / Tool Contract

给 agent 暴露什么动作空间、什么观测格式、什么工具 schema 最有利？SWE-agent 已经证明 ACI 设计会显著影响性能。

1. RL / Post-training for Agents

长轨迹、稀疏奖励、多工具、多 agent、非平稳环境里怎么做 credit assignment、policy improvement、offline/online 结合？Agent Lightning 和 MLGym 是你现在最该盯的入口。

1. Benchmark / Eval

怎么衡量“一个 agent 真的更好”而不是“看起来更聪明”？Anthropic 2026 明确把 eval harness、task、trial、grader、outcome 分开讲了；AgentBench / WebArena / OSWorld / AppWorld / GAIA / MLGym 都属于这条主线。

1. Multi-agent / Managed Agents

什么时候需要多 agent？怎么拆工？怎么防通信开销和 coordination collapse？Anthropic 2025 讲 research system 时就发现，多 agent 特别适合高并行、超单一上下文容量、复杂工具空间的问题。

1. Safety / Governance / Robustness

权限、环境隔离、工具越权、prompt/tool injection、数据外泄、策略漂移怎么防？这个方向现在论文和工程都在快速长，学术价值一点也不低。Anthropic 把“安全、autonomy、eval”连续写成系列，不是偶然。

##### 学习清单升级

**1. Claude / Claude Code**  
保留，但目标从“认证”改成“学 harness”。官方当前能确认的是 Anthropic Academy / Claude Courses 的课程与 completion certificates；我没有在 Anthropic 官方站点核到一个明确名为 “Claude Certified Architect” 的官方证书，所以别把时间过多花在搜外部认证包装上。先吃透官方课程和 engineering blog，收益更大。

**2. Harness design 那篇**  
保留，但给它正确定位：**它不是论文，是系统设计经验文；读法应该是“抽 failure mode 和 design pattern”，不是找数学定理。** Anthropic 2025/2026 这几篇一起读，价值会大很多。

**3. NVIDIA 认证**  
可以保留。NVIDIA 现在官方确实有 **NVIDIA-Certified Professional Agentic AI (NCP-AAI)**，定位是 intermediate，覆盖架构、开发、部署、治理、多 agent、distributed reasoning、scalability 和 ethical safeguards；还有 **NCP-GENL** 和入门级 **NCA-GENL**。如果你想要一个系统性 syllabus，这条线是有价值的。

**4. 源码学习**  
我建议源码顺序改成：  
 先看 Claude Code / Anthropic docs 的设计逻辑  
 → 再看一个开源 agent runtime  
 → 最后横向对比 OpenClaw / Goose / 其他工具。  
因为先有设计轴线，再看实现才不会只剩工程细节。

**5. agent infra**  
可以，但别太早。如果前面没学清楚 memory、context、ACI、eval，你去看 k8s 上的 agent infra 很容易只学到“怎么把一个脆弱系统部署得很稳定”，而不是“怎么把系统本身做对”。

**附：一个更像“研究生入门”的 8 周路线**

- **第 1–2 周：建立系统观**  
  读 Anthropic 三件套：effective agents、context engineering、long-running harness；同时补 ReAct / Reflexion / Generative Agents 的摘要级理解。

- **第 3–4 周：集中读 benchmark**  
  AgentBench、WebArena、GAIA、OSWorld、AppWorld、MLGym。目标不是背结果，而是回答：任务空间是什么？环境如何构建？grader 怎么写？真实度和可复现性怎么平衡？

- **第 5–6 周：集中读 memory / workflow / interface**  
  MemGPT、HiAgent、SWE-agent、WorFBench、AFlow。读完你会明显感觉“Agent 并不是把 prompt 写长一点”。

- **第 7–8 周：读 training / RL，并做一个小复现**  
  至少读 Agent Lightning；如果你想冲科研，最好选一个 benchmark 上复现一个 tiny baseline，比如在 WebArena / OSWorld / AppWorld 上比较不同记忆策略，或者在 workflow benchmark 上比较不同 planning 结构。


##### 最后一句最重要的话

你现在最需要建立的，不是“我会不会用某个 agent 产品”，而是这个判断：

> **Agent 研究的核心不在“聊天更像人”，而在“让模型作为一个带状态、带工具、带环境反馈、带长期目标的决策系统持续工作”。**

一旦你按这个角度看，开发和算法就不再是两张皮了：
**开发是在实现 agent system，算法是在研究这个 system 为什么有效、哪里失效、怎么变得更可学、更可控、更可评、更可部署。** 这恰恰是现在 Agentic AI 最值得学术化的地方。

##### 先说结论：你现在的项目，不是“离 AI Agent 很远”，而是叙事角度错了

你现在的 StablePay / agentpay，其实天然更适合包装成：

**面向 AI Agent 生态的支付运行时、工具调用、安全策略与交易验证基础设施**

而不是：

**一个普通区块链支付微服务**

因为你现有材料里已经明确有这些 Agent 相关元素：

- 产品目标就是服务 **AI Agent 用户** 和 **Skill 开发者**，围绕 OpenClaw 中的付费 skill 购买展开，交互方式是对话式，而不是传统 Web 表单。
- 核心业务是 **Agent 访问 skill → 返回 HTTP 402 → Agent 签名支付 → 链上结算 → 再次访问 skill**，这本质上已经是一个 agentic workflow。
- 技术上你们不是单体脚本，而是有 DID、Payment、Verification、Query、Blockchain Adapter、API Gateway 的微服务拆分，以及 RPC、Redis、RocketMQ、异步补偿、幂等、重试。
- OpenClaw 插件侧已经有一组“工具化能力”：本地钱包创建、DID 注册、支付限额、支付策略、签名、执行付费 skill demo、网关支付等，这些非常接近今天 JD 里说的 custom tools / tool calling / agent runtime。

所以你不是没有 Agent 项目，而是还没把它讲成“**Agent runtime + tool integration + payment guardrails + verification infrastructure**”。

##### 当前 AI Agent 岗位 JD 的升级趋势

我联网补了一圈，现在 AI Agent 岗位 JD 里，常见要求已经明显升级了。

你最开始总结的“LangGraph / LangChain / AutoGen / LlamaIndex / CrewAI + ReAct / RAG / Function Calling + MCP + 向量库”，只覆盖了一部分。最近岗位更常出现的是下面这组：

- **agent orchestration / workflow design**：不仅是调模型，而是设计 agent 的编排逻辑、状态流转和复杂工作流。AppZen 明确写了 *tool calling, orchestration frameworks, advanced context management*；CI&T 直接写了 *architect the agent orchestration logic*。
- **MCP / external tool integration**：CI&T 直接把 *implement tool integrations via MCP* 写进职责。
- **guardrails / human-in-the-loop / safe tool execution**：CI&T 提到 Bedrock Guardrails；Infinity 的岗位写了 *guardrails for safe tool execution* 和 *human-in-the-loop where needed*。
- **observability / debugging / traceability**：CI&T 要求建立 observability 基础设施；LangChain 的 LangSmith 岗位就是专门做 *observability, testing, debugging*；OpenAI 官方文档则把 traces、graders、datasets、eval runs 直接定义为 agent workflow 的核心评测手段。
- **evals**：现在不只是“跑起来”，而是要能系统评估。OpenAI 官方 agent evals 页面就是明证。
- **long-running reliability / production execution**：OpenAI Codex Core Agents 的岗位强调的是让模型能安全执行代码、使用工具、完成 long-running tasks，并且在生产中高可靠运行。
- **stateful agents / memory / retrieval**：Infinity 的岗位明确写了 *stateful agent systems (workflows, tool calling, memory, retrieval)*；Binance 的岗位写了 *task decomposition, agent orchestration, memory sharing and tool-calling workflows*。
- **RAG / vector DB 仍然常见，但不再是全部**：AHEAD 的岗位同时要 RAG、vector DB、evaluation sets、guardrails；Binance 也把多源知识库上的 RAG 作为一部分。

所以现在更准确的说法是：

> **主流 AI Agent JD 已经从“会调 LLM 框架”升级到“会做 agent 系统工程”：编排、工具协议、状态、评测、可观测、安全、长任务可靠性。**

##### 结合你现在的业务，一个明确的分类

下面我按你要求分成：**明确是 / 可能是 / 可能不行 / 明确不行**。

###### 明确是

**1）把 StablePay 做成 MCP Server**  
这是最顺手、也最能打的增强。  
你现在已经有支付、余额、验证、收益、DID、签名这些 API 和插件工具；把它们重新暴露成标准化 MCP tools，就能直接对应 JD 里的 *MCP/tool integration/custom tools*。而且这不是硬蹭热点，它天然服务你的业务：让 LangGraph / OpenAI Agents / Claude / 其他 agent 客户端都能接入 StablePay 支付能力。你们现有产品里已经有支付、验证、余额、收益 API，以及插件侧的签名/支付/策略工具，这些非常适合被重组为 MCP 工具集合。

建议暴露这些 MCP tools：

- `create_wallet`
- `register_did`
- `get_balance`
- `verify_purchase`
- `pay_skill`
- `get_revenue`
- `get_sales`
- `build_payment_policy`

**2）补一个 LangGraph 编排 Demo**  
这个也非常合适。  
做一个“Paid Skill Orchestrator”：

`discover skill -> verify purchase -> if not purchased then quote -> policy check -> human approval if needed -> sign -> pay -> re-execute -> summarize result`

这会把你现在已有的 HTTP 402 支付链路，升级成一个**显式的 agent workflow**，能非常自然地对齐 JD 里的 orchestration / context management / multi-step decision。你们当前 PRD 和技术方案里已经有自动购买阈值、超阈值确认、余额查询、验证 API、支付 API，这些足够支撑一个完整图式流程。

**3）补 agent evals + traces + replay**  
这项非常值钱，而且现在 JD 很吃。  
不是只做“支付能不能成功”，而是做一个 eval harness：

- case 1：余额不足
- case 2：已购买再访问
- case 3：超阈值需要人工确认
- case 4：签名过期 / nonce 重放
- case 5：网关超时 + 重试
- case 6：链上成功但记录延迟，需要补偿

每次 run 记录：

- 输入任务
- tool call 序列
- 状态流转
- 最终 outcome
- 失败类别
- 重放 trace

这会直接对齐现在岗位里高频出现的 observability / evals / debugging。你们现有系统本来就有重试、熔断、消息补偿、幂等、验证日志这些基础，天然适合接一层 trace + eval。

**4）把“支付限额 + DID 签名 + 二次确认”升级成 guardrails / policy engine**  
这不是包装，是你们业务本来就有的东西，只是你要换一种语言讲。  
你们 PRD 里已经有本地支付限额、自动购买阈值、超阈值确认、签名认证、防重放、限流、权限控制。把这套东西重命名成：

- transaction policy
- spending guardrail
- human-in-the-loop approval
- tool execution policy
- risk controls

这会非常贴近 Agent 岗位里讲的 safe tool execution / guardrails。

###### 可能是

**5）加一个向量库 / RAG 层**  
这个**可能是**，不是必加。  
只有当你真有一批需要检索的文档时，它才有意义，比如：

- skill 元数据
- 付费规则 / developer docs
- 支付帮助文档
- 风险与合规规则
- 支付失败排障 runbook

然后做：

- `skill search`
- `policy retrieval`
- `support copilot`
- `developer integration assistant`

如果你只是为了写简历硬上 pgvector / Qdrant，而没有真实知识库和检索场景，这一层会很空。现在 JD 里向量库和 RAG 还常见，但更多是“有真实知识检索需求时”。

**6）多 Agent 拆分：buyer / verifier / risk / settlement**  
这也**可能是**。  
从系统设计角度很漂亮，但你要看规模。你们现在业务还在早期，如果上来就四五个 agent，很容易变成 demo 感很强、业务价值很弱。  
比较合理的做法是**最多两到三个角色**：

- buyer agent：发起购买
- policy/risk agent：判断是否允许支付
- execution agent：签名与执行支付

有业务解释就行，别为了“multi-agent”而 multi-agent。近期 JD 里确实会提 task decomposition / memory sharing / multi-agent，但一般是在复杂场景下。

**7）记忆层：用户消费偏好 / 开发者偏好 / skill profile**  
这个也**可能是**。  
你可以把它做成：

- 用户侧：默认币种、单次限额、自动购买阈值、风险偏好、历史消费摘要
- 开发者侧：常用 skill、常看的收益指标、常用筛选条件

这和 stateful agent / memory 是能对上的。你们当前已有本地支付配置和用户/开发者双身份逻辑，往前走一步就能变成“profile + memory”。

###### 可能不行

**8）为了关键词，强行把项目改成 CrewAI / AutoGen**  
这通常**可能不行**。  
框架名现在还是会出现在 JD 里，但它们越来越像“例子”，不是本体。岗位更看你会不会 orchestration、tool calling、evals、guardrails、production reliability，而不是你能不能背出 5 个框架名。Binance 这类岗位会写 LangGraph / CrewAI / AutoGen，但它同时强调的是 orchestration、memory sharing、tool-calling workflows。

所以：

- 你如果真的做了 LangGraph demo，写 LangGraph，合理。
- 你如果没用过 CrewAI / AutoGen，只是想凑词，反而容易面试暴露。

**9）上来就做 RL / agent fine-tuning**  
这也**可能不行**。  
不是因为它不前沿，而是因为和你现在手头业务不贴，且数据、评测、训练闭环都没准备好。现在官方确实在推 agent RFT / RL，但那是建立在成熟 eval 和 production data 之上的。对你现在最划算的顺序是：**先 eval、再 traces、再考虑训练。**

**10）做很重的多模态 / 语音 agent**  
如果你的核心业务还是 agent pay / paid skill access，这个优先级也不高。  
它可以做，但不是你拿 Agent 岗最短路径。

###### 明确不行

**11）把“支付微服务”直接说成“我做了 AI Agent 平台”**  
明确不行。  
你确实在做 Agent 生态里的一个关键基础设施，但这和“我做了完整 agent platform”不是一回事。

**12）把没做的 MCP / LangGraph / vector DB / evals 写成已做**  
明确不行。  
你可以设计、可以补做、可以在未来两三周内快速落地；但没做过就不能写成 done。

**13）把独立补做项目伪装成公司正式交付**  
明确不行。  
正确写法应该是：

- Internship core project
- Independent extension around internship domain

###### 产品定位

在现有 StablePay API 与 OpenClaw 插件基础上，新增一个**面向主流 agent 框架的支付运行时 SDK**，让外部 agent 应用可以通过 MCP / REST / orchestration workflow 接入 StablePay 的支付、验证、查询与风控能力。

- **Agent 应用开发者**：使用 LangGraph / OpenAI Agents / Claude / 自建 agent runtime 的开发者
- **Agent 平台工程师**：需要可观测、可评估、安全可控的支付工具链

###### MVP 场景

1. 开发者在 LangGraph 应用中调用 `verify_purchase -> pay_skill -> re-execute`
2. 外部 agent 通过 MCP 连接 StablePay，获得余额、验证、支付、收益工具
3. 运维/开发通过 traces + evals 查看某次支付工作流为何失败
4. 策略层自动判断是否允许 agent 支付，必要时触发人工确认

###### MVP 功能

- MCP Server
- LangGraph reference workflow
- Eval harness + scenario dataset
- Tool-call tracing + replay
- Policy/guardrail engine
- 可选：developer docs / support docs 检索层

###### 成功指标

- 第三方 agent 应用接入时间 < 30 分钟
- 支付工作流 E2E 成功率
- 超阈值支付全部经过 HIL
- 关键失败场景可被 eval 覆盖
- 每次支付链路可 trace / replay

**1. `stablepay-mcp-server`**

- `create_wallet`
- `register_did`
- `verify_purchase`

**2. `agent-workflow-demo`**

- 节点 A：discover / quote

- already purchased

输出：

- outcome

只在下面任一成立时再加：

- 你要做 developer support copilot

因为这是现在很多候选人最薄弱、但岗位越来越重视的部分。

#### 查重
1.学生查重通过什么平台进行？
新本科教学管理信息服务平台（zdbk.zju.edu.cn）上线后，已将知网查重功能链入平台，学生可直接通过教务系统进行毕业论文查重。
2.学院需要提前进行什么操作？
学院在毕业论文（设计）——过程管理——论文查重设置——面向学院设置页面，点击右上角“增加”按钮，在弹出框内（如下图），设置查重对应年级（包括延毕学生）、查重起止时间、自动通过的查重率、以及查重率类型（查重率类型包括文字复制比、去除本人文字复制比、去除引用文字复制比3类，学院可根据实际情况自行设置）。
3.学生如何查重？
学院设置好后，学生在毕业论文（设计）——学生过程资料，上传“查重稿（论文初稿）”。确认无误后，进入毕业论文（设计）——论文查重页面后，会出现最新上传的查重稿过程资料。点击“上传查重网站”按钮，查重稿即上传至知网。上传成功后，点击“检测”按钮，进行查重检测。检测结束即可看到查重结果并下载对应查重报告（包括查重报告、全文报告单、简洁报告单、对照报告单）。
4.学生有几次查重机会？查重不通过怎么办？
原则上每位学生有1次查重机会，为避免学生提交的查重报告和上传教务系统的查重稿件不匹配，学生查重通过后暂不能继续在教务系统上传查重稿。
若学生查重不通过，学院在毕业论文（设计）——过程管理——论文查重设置——面向学生设置页面，点击右上角“增加”或“导入”按钮，设置需要2次查重的学生，“查重次数”一栏填2。学院设置好后，学生再次上传修改后的查重稿进行查重。
5.学生点击“上传查重网站”后，始终显示“检测”怎么办？
请学生再次点击“检测状态”下的红色“检测”按钮，一般即可正常显示查重结果。
若学生查重页面有多个记录，须点击最下面一条记录进行上传和检测。
6.学生查重出现“原因包括论文格式、文本加密、制作过程采用特殊工具转换、使用非正版office软件等”报错提示（如下图）怎么办？
出现此提示一般说明上传的查重稿格式有问题，包括论文格式、文本加密、制作过程采用特殊工具转换、使用盗版office等。
请学生尝试新建1个docx文档，将之前文档的内容复制粘贴过来，去除格式保存后再上传检测；如果是pdf可以在转换后把内容复制到word中看下是否有乱码，没有再上传，如果转换后的pdf不能复制内容就需要换一个转换工具（比如：wps或word）进行转换。
该情况不计查重次数，请将调整好的文件重新上传到“查重稿”，再到查重页面查重。
