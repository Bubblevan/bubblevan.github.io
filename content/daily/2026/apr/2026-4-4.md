#### 绘图 x402 flow
一、当前架构与 X402 角色的对应关系

┌─────────────┬─────────────────────────────────────────────────────────────────────────────────────────┐
│  X402 角色  │                                     StablePay 对应                                      │
├─────────────┼─────────────────────────────────────────────────────────────────────────────────────────┤
│ CLIENT      │ 外部 AI Agent（调用方）                                                                 │
├─────────────┼─────────────────────────────────────────────────────────────────────────────────────────┤
│ SERVER      │ 业务服务方（接入 X402 的第三方 API，或你们自己的 demo）                                 │
├─────────────┼─────────────────────────────────────────────────────────────────────────────────────────┤
│ FACILITATOR │ API Gateway + DID Service + Payment Service + Blockchain Adapter + Verification Service │
├─────────────┼─────────────────────────────────────────────────────────────────────────────────────────┤
│ BLOCKCHAIN  │ Solana 链（USDC 合约）                                                                  │
└─────────────┴─────────────────────────────────────────────────────────────────────────────────────────┘

问题的核心在于：FACILITATOR 在图里是一列，但在你们项目里对应的是 6 个微服务的协作链路。下面按 X402 的 12 步逐一展开。

二、FACILITATOR 展开后的 12 步详细映射

阶段 1：初始请求 & 402 触发（步骤 1-3）

步骤 1（CLIENT → SERVER）：这一步与 FACILITATOR 无关，是外部 AI Agent 调用某个 API 服务。

步骤 2（SERVER → CLIENT 返回 402）：SERVER 需要生成 402 响应，附带支付元数据。
目前你们有 GET /api/v1/pay/require 端点，这个端点返回支付要求（定价、收款地址、代币类型）。问题是：这个端点现在是独立的，需要改造成 SERVER 在检测到请求无支付凭证时自动触发的机制。具体来说，API Gateway 需要新增一个中间件：检测请求头中是否携带 X-PAYMENT，若无则代替 SERVER 返回 402，响应体包含从 Payment Service 查回的定价配置。

步骤 3（CLIENT 内部）：CLIENT 自己生成 Ed25519 签名的支付 payload（授权签名，不是真实转账），这步在 AI Agent 侧，与 StablePay 无关。

---
阶段 2：带支付重试 & 验证（步骤 4-6）—— FACILITATOR 核心第一阶段

步骤 4（CLIENT → SERVER）：CLIENT 重发请求，Header 里带 X-PAYMENT: <base64 payload>。
这个请求打到你们的 API Gateway，Gateway 当前的 auth 中间件识别的是 DID-Signature 这类鉴权，需要新增一条路径：识别 X-PAYMENT 头，将其解析为 X402 支付凭证，进入支付验证流程，而不是走普通的 DID 鉴权。

步骤 5（SERVER → FACILITATOR /verify）：这一步在你们微服务里展开为两个串行调用：

1. API Gateway → DID Service（Kitex RPC: VerifySignature）
DID Service 负责验证 X-PAYMENT payload 里的 Ed25519 签名：确认是该 Agent DID 对应的私钥签的，且签名内容（金额、收款方、nonce、timestamp）与请求一致。这一步对应 DID Service 现有的
VerifySignature RPC，但需要扩展：当前签名规范是针对 HTTP 请求本身的，X402 的签名 payload 格式不同（签的是支付授权而非 HTTP 请求），需要在 DID Service 里新增 VerifyPaymentAuthorization 方法。
2. API Gateway/Payment Service → Payment Service（HTTP or RPC: VerifyPayment）
Payment Service 做业务层的支付有效性校验：金额是否在限额内（max_amount_minor 配置）、货币是否支持（USDC/USDT）、idempotency key 是否已用过、nonce 是否重放。
当前问题：Payment Service 的 InitiatePayment 是一个把"验证+结算"合并在一起的操作，X402 要求把这两个阶段拆开——先 verify（快速，同步，不上链），再 settle（后置，可异步）。需要将 Payment Service 拆分出独立的 VerifyPaymentCredential 接口。

步骤 6（FACILITATOR → SERVER 返回验证通过）：Payment Service 将 verify 结果返回给 API Gateway，API Gateway 继续向 SERVER（或直接执行业务逻辑）放行请求。

---
阶段 3：业务执行 & 链上结算（步骤 7-11）—— FACILITATOR 核心第二阶段

步骤 7（SERVER 执行业务）：Server 执行实际业务（AI 计算、返回数据等），这一步 FACILITATOR 不介入。

步骤 8（SERVER → FACILITATOR /settle）：Server 执行完业务后，通知 FACILITATOR 发起结算。
在你们微服务里展开为：SERVER 调用 Payment Service 的 SettlePayment 接口（需新增），传入之前 verify 阶段拿到的 tx_id / payment_id。Payment Service 将该支付记录从 VERIFIED 状态流转到 SETTLING。

步骤 9（FACILITATOR → BLOCKCHAIN 提交交易）：
Payment Service 调用 Blockchain Adapter（Kitex RPC: TransferStableCoin）。这里有一个关键的架构差异需要注意：

- 当前设计：Blockchain Adapter 用的是 StablePay 自己的 Hot
Wallet，以平台的名义向链上发起转账，资金模式是"先预存，后结算"。
- X402 标准设计：CLIENT 在步骤 3 生成的是一个预签名的链上授权（类似 EIP-2612 Permit，或 Solana 的 durable nonce transaction），FACILITATOR 拿着这个客户端自己签的交易去链上广播，资金从 CLIENT 的钱包直接走，FACILITATOR 只是广播者。

这是你们当前架构与 X402 标准最大的结构性差距。有两种改造方向：
1. 保持现有 Hot Wallet 模式（改动小）：FACILITATOR 验证支付授权后，用自己的 Hot Wallet 执行实际转账，但需要在 verify 阶段就确认 Client 有足够余额并锁定额度。
2. 实现客户端预签名广播（更接近 X402 标准）：Blockchain Adapter 新增 BroadcastSignedTransaction 方法，接收 CLIENT 预签名的 Solana 交易并广播，无需 Hot Wallet 介入。

Blockchain Adapter 向 Solana 提交交易（调用 USDC SPL Token 合约的 transfer 或 transferChecked 指令），返回 tx_hash。

步骤 10（BLOCKCHAIN → FACILITATOR 确认）：
Blockchain Adapter 目前有 TxStatusQueryService，会轮询 Solana RPC 查交易状态（PENDING →CONFIRMED/FAILED）。这个机制是现成的，但轮询间隔和超时策略需要根据 X402 的响应时间要求调整。确认后 Blockchain Adapter 将状态返回给 Payment Service（当前是 Payment Service 自己轮询，也可改为 Blockchain Adapter 主动 callback）。

步骤 11（FACILITATOR → SERVER 返回 Settled）：
这里在你们系统里分两条路：

1. 同步路径：Payment Service 等待 Blockchain Adapter 确认后，直接将 Settled 结果返回给
SERVER（适合对响应延迟要求低的场景）。
2. 异步路径（当前架构）：Payment Service 链上确认后，向 RocketMQ 发布 payment_events 消息，Verification Service 消费该消息，更新购买关系表，Query Service 同步数据到分析库。SERVER 通过轮询或 webhook 回调拿到结算结果。

---
阶段 4：返回响应（步骤 12）

步骤 12（SERVER → CLIENT，带 X-PAYMENT-RESPONSE 头）：
SERVER 收到 Settled 通知后，向 CLIENT 返回 200 响应，同时在 Response Header 里加 X-PAYMENT-RESPONSE（包含 tx_hash、settled_at 等）。
这个 Header 的内容由 Payment Service 在 SettlePayment 完成后提供，SERVER 负责透传。当前 API Gateway 没有这个 Header 的写入逻辑，需要新增。

三、展开后的微服务分工汇总
```
X402 FACILITATOR /verify  →
API Gateway（中间件：X-PAYMENT 头解析、nonce 防重放）
    ↓ Kitex RPC
DID Service（VerifyPaymentAuthorization：验签名、验授权内容）
    ↓ Kitex RPC / HTTP
Payment Service（VerifyPaymentCredential：验金额、验幂等、验状态）
    ↓ 返回 verified + payment_id

X402 FACILITATOR /settle  →
Payment Service（SettlePayment：触发结算，状态流转 VERIFIED→SETTLING）
    ↓ Kitex RPC
Blockchain Adapter（TransferStableCoin 或 BroadcastSignedTx：上链）
    ↓ Solana RPC
Blockchain（USDC 合约执行转账，返回 tx_hash）
    ↑ 轮询/callback
Blockchain Adapter（TxStatusQueryService：确认 CONFIRMED）
    ↑
Payment Service（状态流转 SETTLING→CONFIRMED，发 MQ 事件）
    ↓ RocketMQ
Verification Service（消费事件，记录购买关系）
Query Service（消费事件，同步分析数据）
```

我们的 API Gateway 是 X402 协议的入口和出口（解析 X-PAYMENT、写 X-PAYMENT-RESPONSE、返回 402）；DID Service 负责签名验证；Payment Service 是整个 FACILITATOR 的业务编排核心（两阶段：verify + settle）；Blockchain Adapter 是链上执行层；Verification Service + Query Service 是结算后的事件驱动消费层。6个服务合在一起，才构成 X402 图里 FACILITATOR 这一列的完整实现。
