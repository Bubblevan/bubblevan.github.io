### 一、问题识别与动机
在 DualVLN 的单引擎 vLLM 部署中，S2 导航规划已经不再是传统的“短 prompt、长 decode”语言生成问题，而是一个典型的**长多模态 prefill、短文本 decode、随后还要追加 latent-prefill** 的推理流水。输入侧包含多帧图像与文本消息，输出侧通常只生成极短的导航指令或像素目标，而 pixel-goal 分支命中后还需要继续基于同一请求上下文提取 latent queries。这意味着，系统的主要开销不再只来自语言解码，而是集中在**图像 transport、视觉编码、多模态 prefill、以及二阶段 latent-prefill** 上。仓库当前实现也明确保留了这条双阶段路径：`step_s2()` 先执行一次 vLLM 生成，再在命中坐标时构造 `bundle` 并进入 `generate_latents_native_prefill(...)`。

在这一背景下，原始的图片 HTTP 传输和 eager 推理路径都会放大额外开销。前者会把图像重复序列化、编码和反序列化；后者则无法充分利用 vLLM V1 在编译与图执行方面的优化能力。因此，我们将优化目标收敛为两条互补路线：其一，重写**图片 transport**，尽量把大体积图像数据从 JSON/base64 路径中剥离出去；其二，尽量让推理主干落在 vLLM V1 的 **`torch.compile` + CUDA Graph** 优化路径上，尤其覆盖多模态编码器与文本主干前向。 ([vLLM][1])

### 二、共享内存图片 HTTP transport

#### 2.1 设计原则

1. **控制面与数据面分离**：HTTP/JSON 只负责传递消息结构和共享内存描述符，不再直接承载图像像素本体。
2. **图片零拷贝近似化**：图像在 client 侧展开为 RGB 原始字节后写入命名共享内存块，server 仅按名称附着读取。
3. **资源生命周期单边负责**：server 只负责消费和关闭本地句柄，不主动删除共享内存；client 在请求结束后统一清理自己创建的共享内存块。这也符合 Python `SharedMemory` 的语义：`close()` 表示当前进程不再访问，`unlink()` 则表示该共享内存对象的最终销毁应由最后的拥有者负责。 ([Python documentation][2])

#### 2.2 具体方案

现有实现保留了两种图片传输模式：`base64` 与 `image_shm`。transport 模式由 `INTERNNAV_DUALVLN_IMAGE_TRANSPORT` 环境变量控制，默认仍兼容 `base64`，但在优化路径下切换到 `image_shm`。在 `image_shm` 模式中，client 为每张图片分配一个带唯一名称的共享内存块，将 `RGB` 图像展开成连续字节写入共享内存，并在消息体中只保留一份轻量元数据：`shm_name`、`shape`、`dtype`、`mode`、`nbytes`，以及消息内的索引位置。这样，HTTP 只运送一小段结构化元信息，而不再传输完整图片内容。

server 在解码消息时，如果遇到 `image_shm` 类型，则按照 `shm_name` 附着到对应共享内存块，校验 `shape/dtype/mode/nbytes` 后直接恢复为 `PIL.Image`。这一侧只进行读取与 `close()`，不负责 `unlink()`，从而避免服务端和客户端对同一块共享内存产生所有权竞争。client 侧则在请求 finally 阶段统一遍历自己创建的句柄，执行 `close()` 与 `unlink()`，把共享内存的生命周期收束在单次请求内。latents 结果并没有改成共享内存，而仍通过 `torch.save + base64` 回到 HTTP JSON 响应中，因此这一轮优化重点是**图片上行**，不是所有 payload 的全链路二进制化。

#### 2.3 路由与接口层保持不变

为了尽量减少上层 agent 与评测脚本改动，服务端仍保留原有的 `/dualvln/step_s2` HTTP 接口，消息体仍是 JSON。不同之处仅在于，图片字段不再总是携带 base64 字符串，而是可携带一份共享内存描述。服务端接口继续在请求进入后完成：JSON 解析、消息解码、`runner.step_s2(...)` 调用、latents 重新编码与响应发送。也就是说，整个优化被封装在 transport 编解码层，未改变上层 DualVLN 的 API 契约。

### 三、`torch.compile` 与 CUDA Graph 优化

#### 3.1 优化目标

共享内存 transport 解决的是**图片上行传输成本**，而 `torch.compile` 与 CUDA Graph 解决的是**服务端模型执行成本**。vLLM 的官方设计中，`torch.compile` 用于将模型前向编译为更高效的执行图；而 CUDA Graph 则用于捕获和复用稳定形状下的 GPU 执行序列，以减少 host 侧 kernel launch 开销。这两者是正交的：compile 关注图与 kernel 优化，CUDA Graph 关注运行时调度与 launch overhead 折叠；在多模态场景下，encoder graphs 与 decoder graphs 也可以同时存在。([vLLM][1])

#### 3.2 在项目中的接入方式

本项目并没有手写任何 CUDA Graph capture 逻辑，而是将服务端推理尽量对齐到 **vLLM V1 的默认优化路径**。具体做法有两点：

第一，`DualVLNSingleVLLMRunner` 在创建 `LLM(...)` 时显式暴露了 `compilation_config`，并允许在服务启动时把 `{"compile_mm_encoder": True}` 传入 vLLM。这样，优化范围不再局限于文本主干，还能够覆盖多模态编码器。vLLM 官方文档明确说明，多模态 encoder compile 默认关闭，需要通过 `compile_mm_encoder: true` 显式开启。 ([vLLM][1])

第二，服务脚本增加了 `--enable-mm-encoder-compile` 与 `--force-v1` 两个控制项。前者决定是否向 runner 注入 `{"compile_mm_encoder": True}`，后者在 engine 初始化前设置 `VLLM_USE_V1=1`，确保单引擎 HTTP server 也能走 vLLM V1 的优化栈，而不是局限在传统 eager 路径。与此同时，`enforce_eager` 仍作为可选参数保留，但默认不再强制打开，从而为 compile 与 graph 路径留出空间。 

#### 3.3 多模态 encoder compile 的角色

在这个任务里，`compile_mm_encoder` 的作用并不是改变 DualVLN 的语义，而是让视觉编码器和其他可编译模块进入与文本主干一致的编译框架。vLLM 官方文档对这一路径的定义非常明确：它通过 `@support_torch_compile`、`set_model_tag(..., is_encoder=True)` 与 `set_forward_context` 等机制，把多模态 encoder 纳入统一的 compilation config 管理；一旦模型侧具备相应装饰和上下文传递，就可以在不改动用户调用方式的前提下，为视觉前向引入编译优化。([vLLM][1])

#### 3.4 CUDA Graph 在该方案中的位置

本项目层面没有新增任何显式 graph capture API，因此 CUDA Graph 优化并不是“我们自己实现了一套图捕获器”，而是**通过进入 vLLM V1 的默认执行模式，让服务端有机会复用其内部的 encoder/decoder graph 机制**。vLLM 官方文档指出，视觉 encoder graph 与 decoder graph 可以同时启用，且对小 batch、较小图像或高频短请求场景尤其有价值，因为这类场景里 host-side kernel launch overhead 更容易成为显性成本。对于导航规划这类持续、高频、请求模式相对稳定的 workload，这种路径天然比 eager 更契合。([vLLM][3])

### 四、整体方法论

综合起来，这套优化不是单点技巧，而是一条围绕 **“长多模态 prefill、短 decode、二阶段 latent-prefill”** 特征展开的系统化重构：

* 在 **transport 层**，把图片从 base64/JSON 负载中剥离，改用共享内存承载大体积像素数据，HTTP 只保留控制信息与描述符。
* 在 **服务层**，保持 `/dualvln/step_s2` 接口和现有 client/server 调用方式不变，使优化对上层 agent 透明。
* 在 **运行时层**，通过 `VLLM_USE_V1`、`compilation_config={"compile_mm_encoder": True}` 与非 eager 执行路径，把推理主干尽量对齐到 vLLM 的 compile 与 CUDA Graph 栈上，而不是在项目内部手写新的执行器。  ([vLLM][1])
* 在 **语义层**，不改动 DualVLN 的请求协议、坐标解析、latent bundle 构造和二阶段 latent-prefill 逻辑，只压缩传输与执行开销，不改变模型输入输出契约。

### 一、问题识别与动机

在DualVLN的导航规划系统中，S1（底层扩散轨迹生成器）运行于 `nextdit_async` 模式。通过性能剖析发现，单次 `generate_traj()` 调用（包含 10 个扩散步）中，**encoder‑side 的条件表征计算存在大量重复**。

具体而言，`hidden_states_input`（由轨迹潜码、异步记忆编码器输出拼接而成，再复制为 cond/null 双分支）在扩散循环开始前就已完全确定，且**在整个去噪过程中保持恒定**。然而，现有 `LuminaNextDiT2DModel` 的每次前向传播都会独立执行：

- `caption_projection(encoder_hidden_states)` —— 将条件张量投影到模型内部维度；
- 每个 `LuminaNextDiTBlock` 中的 `norm1_context(encoder_hidden_states)` —— 对条件做层归一化。

这两类计算与当前时间步 `t`、当前噪声潜码 `latents` 完全无关，属于**纯静态的 encoder‑side 预处理**。在 10 个扩散步、12 层 transformer 的配置下，相同计算被冗余执行了 120 次以上，成为 S1 推理延迟的重要贡献因素。

**核心洞察**：这是精确等价的冗余计算，而非近似推理的机会。因此，最适合的优化手段是 **exact caching**，而非跳步或复用隐藏状态。

### 二、解决方案设计

#### 2.1 设计原则

1. **语义不变性**：缓存开启时，输出张量与关闭时严格一致（仅允许浮点舍入误差）。
2. **局部性**：缓存生命周期限定于**单次 `generate_traj()` 调用**，不跨 episode 或请求共享，避免状态污染。
3. **模块隔离**：只修改 S1 相关文件（`system1_runner.py`, `nextdit_traj.py`, `nextdit_crossattn_traj.py`），不动 S2 / vLLM / 请求调度层。
4. **可度量性**：新增专用 metrics 字段，区分于原有的 `diffusion_steps_reused`（后者语义为“跳步”）。

#### 2.2 缓存内容（第一版）

- **投影缓存**：`caption_projection(encoder_hidden_states)` 的结果，在 `LuminaNextDiT2DModel.forward()` 入口处复用。
- **逐层归一化缓存**：每个 `LuminaNextDiTBlock` 中 `norm1_context(encoder_hidden_states)` 的输出，按层索引存储。

这两项缓存都是 **exact** 的：不依赖 `timestep`、`latents`、`attention mask` 等动态量。

#### 2.3 可选扩展（这个还没做！） 

- **Cross‑attention K/V 缓存**：缓存每层 cross‑attention 中由 `encoder_hidden_states` 计算得到的 **Key 和 Value 投影**。该缓存同样精确，但需要侵入 `diffusers.Attention` 内部，实现复杂度更高，因此作为第二版实验开关，默认关闭。

### 三、具体改动路径

#### 3.1 控制流改造（`system1_runner.py`）

- 为 `generate_traj()` 增加两个布尔开关：  
  `dit_cond_cache_enabled`（第一版缓存）和 `dit_crossattn_kv_cache_enabled`（实验性）。
- 将原有构建 `hidden_states_input` 的逻辑封装为辅助函数 `_build_hidden_states_input()`，并在扩散循环**之前**，若缓存开启则调用 `_build_dit_cond_cache()` 预计算静态条件。
- 缓存结构以字典形式传递：包含 `"projected_encoder_hidden_states"` 和 `"layer_normed_encoder_hidden_states"`（列表，长度等于层数）。
- 在每次调用 `self.traj_dit(...)` 时，将 `cond_cache` 作为关键字参数传入。

#### 3.2 接口透传（`nextdit_crossattn_traj.py`）

- `NextDiTCrossAttn.forward()` 增加可选参数 `cond_cache=None`，直接透传给底层 `LuminaNextDiT2DModel.forward()`。
- 不在此层做任何实质性缓存逻辑，保持薄包装。

#### 3.3 核心缓存逻辑（`nextdit_traj.py`）

- `LuminaNextDiT2DModel.forward()` 增加 `cond_cache` 参数。
- **投影缓存**：  
  - 若 `cond_cache` 中存在 `"projected_encoder_hidden_states"`，则直接赋值给 `encoder_hidden_states_proj`，跳过 `self.caption_projection(encoder_hidden_states)`。
  - 否则正常计算，并视开关状态写回缓存。
- **逐层归一化缓存**：  
  - 遍历 `self.transformer_blocks` 时，检查 `cond_cache` 中对应层的 `layer_normed_encoder_hidden_states[i]`。
  - 若存在，则直接作为 `norm1_context` 的输出；否则现场计算并可选写回。
- **K/V 缓存（这个还没做！）**：   
  - 当 `dit_crossattn_kv_cache_enabled=True` 时，在每层 cross‑attention 的 `Attention` 模块中，若 `cond_cache` 包含该层的预计算 K/V，则直接使用；否则计算后缓存。  
  - 注意：K/V 缓存仍只复用 encoder‑side 投影，query 保持动态。

所有缓存写入操作均需确保**不改变原始张量的梯度历史**（使用 `.detach()` 或保持 `requires_grad` 状态与原始一致）。

#### 3.4 同步（可选，`internvla_n1.py`）

- 若存在其他调用 `generate_traj()` 的路径（如训练脚本），建议将相同的缓存开关和透传逻辑同步至此，以保证行为一致。第一版优先保证 `system1_runner.py` 路径可用。

## 一、问题识别与动机

在 DualVLN 的双系统导航架构中，S2 负责视觉语言规划；当 S2 生成出 `pixel goal` 时，系统还需要进一步提取与该规划结果对应的 `latent goal`，再将其交给 S1 的轨迹生成器使用。现有实现中，这一 latent 提取并不是在原始文本生成过程中顺带完成的，而是在文本生成结束后，额外构造一条新的 latent request：将 `prompt_token_ids`、`generated_token_ids` 与若干个 `<TRAJ>` token 重新拼接成新的 prefill 序列，再执行一次 native latent prefill，并取最后 `n_query` 个 hidden states 作为 latent 表征。

这一实现方式带来了一个结构性冗余：对于每一次成功产出 `pixel goal` 的 S2 调用，系统都会在**同一份多模态上下文**上执行两次请求级前向。第一次用于正常文本生成；第二次用于 `<TRAJ>` 后缀的 latent 提取。由于导航场景中的 S2 推理是典型的 **long prefill, short decode** 模式，第二次 latent prefill 并不是一个轻量补丁，而会重复经过 prompt 组织、多模态元数据对齐、调度与前向执行，从而成为 S2 平均延迟的重要组成部分。

**核心洞察**在于：这里真正冗余的，不是文本 decode 本身，而是“文本结束后再为 latent 提取重走一次请求级 prefill”。因此，更合适的优化方向不是额外引入近似缓存或跨步复用，而是尽量保持原 latent 定义不变，将第二次 latent 提取从“独立 synthetic request”改造成“同一 live request 上的短后缀 continuation”。

## 二、解决方案设计

### 2.1 设计原则

1. **语义保持**：latent 的定义仍然保持为“文本生成完成后，追加 `<TRAJ>` suffix，并读取最后 `n_query` 个 hidden states”，不改训练语义，不改 S1 消费接口。
2. **不破坏公开文本 API**：不修改 vLLM 公共 `generate()` / `chat()` 的返回格式，只增加一条 DualVLN 专用的 internal-only continuation 路径。 
3. **复用 live request 状态**：continuation 仅作用于当前已经完成文本生成、但尚未彻底释放的 request，复用其 request-local KV / block / batch 状态，而不是重新构造第二条请求。 
4. **失败可回退**：任何不满足安全条件的情形，都显式退回旧的 native latent prefill 路径，保证系统可用性。 

### 2.2 优化对象

本次优化针对的不是整个 S2 文本生成，而是其中**“文本生成完成后，为 latent 提取而额外执行的第二次 prefill”**。旧路径中，这一阶段需要：

* 重新构造 `prefill_token_ids`
* 重新附着 multimodal metadata
* 重新通过 engine / scheduler / worker 执行 latent prefill
* 再取后缀 hidden states 作为 latent

新路径的目标，是把这一步改造成：

* 在文本 request 自身上挂一个 continuation intent
* 在 request 停止、但尚未被 free 的窄窗口中，直接为 `<TRAJ>` suffix 追加 KV block
* 在同一个 live request 上继续一个极短的 suffix forward
* 直接收集 suffix 对应的 hidden states 作为 latent 结果。   

### 2.3 方法边界

这里复用的是**同一请求内部已经存在的 KV / prefix 状态**，因此它与“跨请求共享缓存”或“S1 那种 encoder-side exact cache”不同。
也正因为如此，我不建议把它表述成“Cross-attention K/V cache”。更准确的说法是：

* **same-request suffix continuation**
* **request-local KV reuse for latent extraction**
* **second-pass latent prefill elimination**

这能更准确反映它的实现边界与贡献对象。

## 三、具体改动路径

### 3.1 DualVLN 入口控制（`dualvln_single_vllm.py`）

`step_s2()` 仍然先执行一次正常的文本生成；不同的是，在需要 latent 的分支中，不再直接进入旧的 bundle + native latent prefill，而是优先尝试 same-request continuation。为此，运行时会记录 continuation 是否启用、是否尝试、是否实际使用，以及失败时的 fallback 原因；只有在 continuation 不可用或执行失败时，才重新退回原先的 latent prefill 路径。

### 3.2 前端请求封装（`llm.py` / `request.py` / `engine/__init__.py`）

在 vLLM 前端层，为 DualVLN 增加了一个内部结构 `DualVLNLatentContinuation`，其内容仅包括：

* `suffix_token_ids`
* `n_query`

该结构不改变公开请求接口，而是作为 internal-only intent 附着在 `EngineCoreRequest` 和 `Request` 上，用于告诉后端：当前请求在文本结束后还需要继续一个很短的 `<TRAJ>` 后缀，并返回其 hidden states。 

### 3.3 调度与 continuation 规划（`scheduler.py`）

当 scheduler 判断一个请求已经正常完成文本生成时，如果该请求携带了 `dualvln_latent_continuation` 信息，就不会立刻将其完全作为普通 finished request 处理，而是先调用 continuation 规划逻辑。该逻辑会检查若干安全条件，例如：

* suffix 是否非空
* `n_query` 是否与 suffix 长度一致
* 是否为单请求场景
* 是否支持当前调度模式与并行模式
* 是否能够为 continuation 追加所需 KV blocks

若这些条件成立，则记录一条待执行的 continuation attempt，并把所需的后缀 token、block 分配结果以及当前 request 状态一起保留下来，供 engine 在文本生成结束后立刻执行。

### 3.4 Engine 执行与结果回传（`core.py`）

在 `scheduler.update_from_output(...)` 之后，EngineCore 会检查是否存在待执行的 DualVLN continuation attempt。若存在，就通过内部 worker RPC 调用 continuation helper，在不重建第二条请求的前提下，直接沿当前 request 的 live state 继续执行 suffix forward，并把得到的 hidden states 暂存为 `DualVLNLatentContinuationResult`。前端随后通过 request id 将这份结果取回，并作为 S2 的 latent 输出；若 continuation 未成功执行，则前端能够得到明确的 fallback reason，并退回旧路径。

### 3.5 Worker 侧 continuation helper（`gpu_worker.py` / `gpu_model_runner.py`）

真正的 suffix continuation forward 在 GPUModelRunner 内完成。该 helper 直接使用当前 request 对应的 `CachedRequestState` 与 `InputBatch` 状态，在已有 request-local KV 的基础上：

* 为当前 request 追加 `<TRAJ>` suffix
* 更新其 token / block / computed token 状态
* 构造一个仅覆盖该 request 的最小 continuation 执行上下文
* 跑一次短后缀 forward
* 收集最后 `n_query` 个 hidden states

这样，latent 提取就从“重建整条 synthetic request 的第二次 prefill”，收缩为“在同一 live request 上继续一个很短的 suffix continuation”。
