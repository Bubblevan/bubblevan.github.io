OWS 的 wallet create 本来就会一次性派生多链地址。具体而言：
- `Wallet created: 071ccfe1-c6c4-4b4c-bbad-274033f17c61`：这是 OWS 的钱包 ID（UUID），不是链上地址。官方把 wallet id 定义成 UUID v4。
- `solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp`：这是 Solana 的 CAIP-2 链标识，不是地址。OWS 用 CAIP 标识链。
- `2gL5tHKBp2ZWGX9chgcHFtEqLcwG2GT76CimizqFqXXF`：这才是 这个 OWS 钱包在 Solana 上的地址 / 公钥。OWS 文档写明 Solana 地址格式就是 Base58 编码公钥。
- `Path: m/44'/501'/0'/0'`：这是 Solana 派生路径。

#### 全局结果
这份 1839 episode 的全样本结果非常支持我的核心叙事：
- 基础导航能力并不差：SR=63.68%，SPL=57.69%，NE=4.08
- 但系统协同明显受限于 S2 时延和 pixel-goal 产出不足
- 问题是结构性的，不是偶发长尾造成的

最关键的是这一组：

- s2_step_latency_ms_mean = 507.98 ms
- s2_step_latency_ms_p50 = 508.86 ms
- s2_step_latency_ms_p90 = 657.28 ms
- s2_step_latency_ms_p99 = 672.00 ms

这说明 S2 不是“偶尔特别慢”，而是“平时就慢”。因为连 p50 都在 509ms，中位数本身就已经很高了。这个结论非常适合你答辩时讲。

|序号|核心结论|关键指标数据|深度分析|
| ---- | ---- | ---- | ---- |
|1|S2延迟为常态高延迟，而非尾部慢延迟|平均延迟508ms，中位数509ms，p90 657ms，p99 672ms|S2推理延迟分布稳定，但整体时延基线偏高，系统优化应优先下压整体时延基线，而非解决极端抖动问题|
|2|双系统协同存在明显不足|pixel_goal_yield_rate=0.3218、s1_trigger_rate=0.3218、s2_discrete_action_rate=0.6782|仅32.2%的S2请求可生成pixel goal并触发S1，67.8%的情况为离散动作，闭环易退化为S2主导的低频控制|
|3|S1能力有效但触发量不足|avg_s1_actions_per_trigger=11.08、avg_s1_rollout_calls_per_trigger=2.77|S1被触发后可稳定实现局部连续控制，并非无效模块；瓶颈在于S2时延高、pixel goal产出少，需优先优化S2|
|4|控制间隙远超平滑控制标准|端到端控制间隙均值322.47ms，p90 730.65ms；人类平滑控制时延150-250ms|控制间隙显著高于平滑闭环控制的时延要求，实际部署中时延问题突出|

#### [DiT Cross-attn KV Cache](https://chatgpt.com/g/g-p-69cf21eda9dc81918912d00bc2007bc3-dualvln/c/69d2b045-9270-8326-bd46-349c6e5a29a9)
在配置文件里这样开启：
```bash
eval_cfg.agent.model_settings["dit_cond_cache_enabled"] = True
eval_cfg.agent.model_settings["dit_crossattn_kv_cache_enabled"] = True
```
指标形状：
- s1_crossattn_kv_cache_enabled = true
- s1_crossattn_kv_cache_hits = 92556
- s1_crossattn_kv_cache_misses = 10284
- s1_crossattn_kv_cache_saved_ms_total = 8616.55

而且这组 hit/miss 和理论值完全对上。如果每次 generate_traj() 是 10 个 diffusion steps + 12 个 cross-attn block：
那么每次 rollout 一共有 10 × 12 = 120 个 layer-step 的 cross-attn K/V 机会。
这次 s1_rollout_call_count = 857，所以总机会数是 857 × 120 = 102840
而现在的：
hits + misses = 92556 + 10284 = 102840
完全对上。
更进一步，如果 K/V cache 的策略是“每次 generate_traj 的第 1 个 step 对每层先算一次 K/V，后面 9 个 step 复用”，那每次调用就应该是：
- miss：12
- hit：9 × 12 = 108
乘上 857 次 rollout：
- miss：857 × 12 = 10284
- hit：857 × 108 = 92556

更重要的是，S1 局部收益也确实更大了一截，相对而言：
- s1_generate_traj_ms_total: 146.12 → 132.33，约 -9.44%
- s1_dit_loop_ms: 135.38 → 121.64，约 -10.15%

这个量级已经比第一版 cond cache 单独带来的 6% 左右更像样了。
所以如果只问 “S1 的 KV cache 有没有真的带来额外收益”，我的答案是：有，而且证据相当强。
