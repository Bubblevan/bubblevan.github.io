### 1. 输入到底是什么？
给**策略（NaVILA）**的输入只有两样：

| 输入 | 来源 | 说明 |
|------|------|------|
| **RGB 图像** | `batch["rgb"]`，当前 + 历史多帧拼成“视频” | 第一人称 512×512，90° FOV，仅 RGB |
| **自然语言指令** | `episode.instruction.instruction_text` | 一句描述目标的话，例如 “Walk down the hall and turn left into the bedroom.” |

代码里策略只用到了这两样（见 `navila_trainer.py` 161–168 行）：  
用 `batch[0]["rgb"]` 做图像，用 `current_episodes[0].instruction.instruction_text` 做指令，没有用 depth，也没有用任何 map。

仿真里还有、但**没有**喂给策略的：

- **Depth**：Habitat 配置里有 `DEPTH_SENSOR`，但 NaVILA 的 eval 里**没有**把 depth 送进模型，只用了 RGB。
- **BEV / 俯视图**：**不提供给策略**。  
  - 只有在保存视频时，为了**可视化**才会算 `TOP_DOWN_MAP_VLNCE` 并画在视频里（`observations_to_image` 里用 `info["top_down_map_vlnce"]`）。  
  - 策略的 forward 里没有任何 map/BEV 输入。
- **目标点坐标**：`episode.goals[0].position` 只用于**评估**（算 success、SPL、到终点的距离）和 oracle（shortest path follower），**不**作为策略输入。

所以：**机器人只知道“当前看到的 RGB + 一句文字指令”，不知道目标在哪个坐标、也没有地图或 BEV。**

### 2. 机器人怎么知道要去哪个房间？

**只能靠“指令 + 看到的场景”做语义匹配。**

- 数据集中每个 episode 有：
  - 起点、终点（`episode.goals`）、
  - 一条**自然语言指令** `instruction_text`，描述从起点到终点该怎么走（例如 “Go to the kitchen past the sofa”）。
- 策略**不知道**：
  - 目标点的 (x, y, z)、
  - 全局地图、
  - 自己在地图上的绝对位置。
- 策略**只知道**：
  - 当前（及历史）第一视角 RGB、
  - 这句指令。

所以“要去哪个房间”完全靠：  
**指令里说的“房间/物体/动作” + 当前视野里看到的场景** 是否匹配；没有显式的“目标房间”的坐标或 BEV。


### 3. 为什么很多 episode 在规定时间内找不到？—— exploration vs exploitation

- **利用（exploitation）**：  
  - 根据当前视野和指令，选“看起来最像在往目标走”的动作（例如“左转进卧室”）。  
  - 若一开始就选错方向或错失关键路口，可能一直在一个区域打转。
- **探索（exploration）**：  
  - 需要主动多走走、多看看，才能发现“原来目标在另一条走廊”。  
  - 但步数上限（如 500 步）、时间/长度限制会卡死，探索太多又可能到不了终点。

具体会表现为：

1. **指令歧义 / 长路径**：指令可能对应多条路径或很长路径，模型容易在某个岔路选错，且没有全局信息纠正。
2. **无全局信息**：没有地图、没有目标坐标，错了只能靠“再走走看”来试，容易浪费步数。
3. **局部最优**：模型可能总在“当前看起来合理”的局部动作上重复（例如一直在同一片区域转），缺乏“先退出来再换一条路”的探索。
4. **步数上限**：R2R/RxR 通常有 `MAX_EPISODE_STEPS`（如 500），探索和回头都会耗步数，时间到了就强制结束，所以会看到很多“没找到指定房间”的 episode。

所以：**在规定时间内找不到指定房间，正是“只给 RGB+指令、不给 BEV/目标坐标”下的典型现象，和 exploration/exploitation 的权衡直接相关。**


### 4. NaVILA 的输出是什么？就是终端里那几句文字

**模型真正的输出**：一段**自然语言文本**，最多 32 个 token（由 `max_new_tokens=32` 限制）。

终端里看到的：

- **第一行句子**（例如 `The next action is turn left 45 degree.`、`I think I should stop because I have finished the instruction.`）  
  → 就是 **NaVILA 的原始输出**，对应代码里的 `print(outputs)`（navila_trainer 第 219 行）。
- **第二行 `[0]` / `[1]` / `[2]` / `[3]`**  
  → 是**从这段文本里用正则解析出来的动作编号**，对应 `print(actions)`（第 240 行）。

所以：**“输出” = 那段英文句子；`[数字]` 是下游用规则从这句话里抽出来的动作。**

### 5. 用给的例子走一遍流程

#### 例子 1：转弯

```
The next action is turn left 45 degree.
[2]
queue length: 2
```

- **模型输出**：整句 `The next action is turn left 45 degree.`
- **动作解析**：
  - 用正则找 `"is turn left"` → 命中 → `actions[0] = 2`（左转）
  - 再正则 `turn left (\d+) degree` → 得到 45
  - 45 按 15 对齐 → 还是 45°，所以会执行 45/15 = **3 次** TURN_LEFT
- **执行方式**：先 `envs.step([2])` 执行 1 次，剩下 2 次放进 `queue_actions`，后面两步会 `using queue...2` 再执行两次。所以终端出现 `queue length: 2`。

#### 例子 2：前进

```
The next action is move forward 75 cm.
[1]
using queue...1
queue length after using...1
...
queue length after using...0
```

- **模型输出**：`The next action is move forward 75 cm.`
- **动作解析**：
  - 正则 `"is move forward"` → `actions[0] = 1`（前进）
  - 正则 `move forward (\d+) cm` → 75
  - 75 按 25 对齐 → 75，即 75/25 = **3 次** MOVE_FORWARD
- **执行**：先 step 1 次，剩下 2 次进 queue，后面两步各 `using queue...1` 执行完。

#### 例子 3：停止（带一点“推理”）

```
I think I should stop because I have finished the instruction.
[0]
```

- **模型输出**：整句 `I think I should stop because I have finished the instruction.`
- **动作解析**：正则里有 `\bstop\b`，在这句话里命中 → `actions[0] = 0`（STOP）。
- **执行**：`envs.step(actions)` 发停止，episode 结束。

所以：**“调动动作前进或转弯”** 就是：**模型只负责生成这句话；代码用正则在这句话里找 “stop / is move forward / is turn left / is turn right” 和数字，再转成环境动作 0/1/2/3 和重复次数。**

### 6. 有没有 CoT（Chain-of-Thought）？

- **没有在代码里强制 CoT 格式**：prompt 只要求“根据图像和指令决定下一个动作”，并说明可以是 turn left/right、move forward 或 stop，**没有**要求“先写推理再写动作”。
- **但模型可以自己先“想”再“说”**：  
  - 像 `I think I should stop because I have finished the instruction.` 就是先一句简短推理，再给出“stop”。  
  - 所以可以理解为：**有“隐式 CoT”（模型自己选择在 32 token 里写推理），但代码不解析 CoT，只解析动作相关的那几个短语和数字。**

也就是说：**CoT 是模型自由发挥的；系统只从整段输出里抠出“动作 + 参数”。**


### 7. 动作空间：只有 4 种，没有“后退”

环境定义的动作（来自 VLN-CE v0）是：

| 动作编号 | 含义           | 解析方式 |
|----------|----------------|----------|
| 0        | STOP           | 文本里出现 `stop` |
| 1        | MOVE_FORWARD   | 文本里出现 `is move forward`，可选解析 `move forward (\d+) cm`（否则默认 25 cm） |
| 2        | TURN_LEFT      | 文本里出现 `is turn left`，可选解析 `turn left (\d+) degree`（否则默认 15°） |
| 3        | TURN_RIGHT     | 文本里出现 `is turn right`，可选解析 `turn right (\d+) degree`（否则默认 15°） |

- **没有“后退”**：R2R/RxR 的 action space 里就没有 backward，所以 NaVILA 也不会输出后退，只能通过“转 180° 再 forward”来等效。
- **前进/转弯的“幅度”**：  
  - 前进：25 / 50 / 75 cm（按 25 取整到最近）。  
  - 转弯：15 / 30 / 45°（按 15 取整）。  
  模型只要在句子里写出数字，代码就按上面规则解析并拆成多步 `step` 或放进 queue。

### 8. 简短总结

- **NaVILA 的输出**：就是终端里那**一行英文**（例如 “The next action is turn left 45 degree.”），即 VLM 的生成文本。
- **`[0]`/`[1]`/`[2]`/`[3]`**：是代码用正则从这段文本里解析出的动作编号。
- **有没有 CoT**：没有强制格式，但模型可以在同一段话里先写理由再写动作（隐式 CoT）；代码不关心推理，只抠“stop / is move forward / is turn left / is turn right”和数字。
- **怎么“调动动作”**：  
  - 模型只负责生成那句话；  
  - 代码用 4 个正则 + 两个数字正则（cm、degree）得到“动作类型 + 距离/角度”；  
  - 再按 25 cm / 15° 拆成多步，用 `envs.step([action])` 和 `queue_actions` 执行，**没有后退，只有停/前进/左转/右转**。
