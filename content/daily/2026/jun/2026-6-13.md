# 从 Pico 到面向大厂 Agent Runtime / Harness 岗位的项目重构报告

## 0. 结论先行

Pico 这类本地 coding harness baseline 的价值，不在于“我接了几个工具、做了几层安全校验、写了几个 benchmark”，而在于它已经有一个最小闭环：CLI 入口、runtime loop、context manager、tool registry、memory、checkpoint、run artifacts、evaluator。它适合作为 v0，而不适合作为最终项目。原始 Pico 文档把它定义为“面向代码仓库长链路任务的本地代码 agent harness”，重点覆盖模型接入、工具调用、上下文管理、任务恢复、结构化记忆、运行审计和评测闭环；这说明它的骨架是完整的，但也说明它非常容易被复用者讲成同一套话术。[Pico]

如果目标是 Agent 开发 / 算法实习，尤其是你贴出来的 DeepSeek Harness / Agent Runtime 类 JD，那么项目不能继续围绕“工具审批、安全边界、人工确认”来包装。大厂不会因为你做了路径逃逸拦截、危险命令弹窗、敏感信息脱敏就觉得你有核心竞争力。这些最多是工程 hygiene。更有价值的方向是：Agent 运行时的可扩展架构、上下文与模型调用链路优化、容器生命周期与执行隔离、并发调度、状态恢复、成本/延迟/成功率的联合优化、真实 coding workflow 的可复现实验。

因此建议将 Pico 重构为：

**RepoPilot Runtime：面向多仓库 Vibe Coding 的容器化 Agent Harness 与评测平台**

项目定位不是“安全 coding agent”，而是：

> 设计并实现一个面向真实代码仓库任务的 Agent Runtime / Harness，支持多模型调用链路、token-aware 上下文管理、容器化工具执行、workspace 生命周期管理、MCP/Skill 工具生态接入、长任务状态恢复、并发任务调度和可复现实验评测；通过自建 RepoTaskBench 评估不同 harness 配置对任务成功率、工具效率、上下文效率、容器冷启动、并发吞吐和成本的影响。

这个定位与 JD 中的三类能力强相关：第一，下一代容器调度与隔离平台；第二，通用 Agent 产品的整体架构和核心模块演进；第三，模型调用链路、上下文处理、提示工程和复杂场景策略设计。安全仍然存在，但它不再是项目主菜，而是容器化 runtime 和 tool execution plane 的副产物。

## 1. 近一年 Agent 研究趋势：从“模型会不会规划”转向“系统怎么让 Agent 稳定执行”

过去 Agent 论文常围绕 ReAct、SWE-agent、WebArena、AutoGPT 这类经典范式展开。但最近一年更值得看的变化是：研究对象从单个 prompt / agent loop，转向了 harness、runtime、tool protocol、memory、benchmark construction、multi-agent coordination、test-time scaling、真实 PR 数据和企业环境评测。换句话说，Agent 研究正在软件系统化。

### 1.1 Harness 正在变成独立研究对象

新的 harness 论文已经不再把 harness 当成普通 glue code，而是把它当成影响 agent 能力的关键变量。Harness-Bench 这类工作直接提出：Agent 能力不应只归因于 base model，而应该报告为 model-harness configuration。它关注 context、tools、state、constraints、permissions、tracing、recovery 等执行层变量如何影响最终表现。[R1]

AutoHarness 更激进，提出“code as harness”：让 LLM 自己生成或补全执行 harness，以适配不同任务环境。这类工作透露出的信号是：harness 不再只是工程实现，而是可以搜索、生成、评估和优化的对象。[R1]

AgentSquare 进一步把 agent 设计拆成 planning、reasoning、tool use、memory 等模块，并做自动化组合搜索。它对你的启发是：不要把 Pico 的 runtime、memory、context、tools 写成固定死结构，而要将它们参数化，变成可 ablation、可搜索、可对照的 harness design space。[R2]

这对项目很重要。你不应该只说“我设计了上下文管理”，而应该说“我将 harness 拆成 Execution Loop、Tool Registry、Context Manager、State Store、Lifecycle Hooks、Evaluation Interface 六类可替换模块，并通过统一任务环境比较不同配置的效果”。这更像 Agent 系统研究，也更像实习岗位中会做的事情。

### 1.2 顶会论文关注点：工具调用、记忆、多智能体、可验证评测

ACL 2025 的 ToolMaker 方向很有参考价值：它研究让 LLM agent 自动从 GitHub URL 和任务描述出发，安装依赖、生成可调用工具，并通过闭环调试修复工具实现。它不是单纯“给 agent 更多工具”，而是研究工具如何被自动构造、适配、验证。[R3]

MultiAgentBench 关注多智能体协作和竞争，不只看 task completion，还引入 milestone-based KPI，比较 star、chain、tree、graph 等 coordination protocol。这意味着多 agent 项目不能只写“我支持子 agent”，而要说清楚拓扑、消息协议、任务拆分、子任务验收、冲突合并和成本控制。[R3]

ToolSandbox、ACEBench、BFCL 等工具评测工作则说明，tool use 的研究已经从“能不能输出函数名和参数”走向多轮、有状态、交互式、真实 API、特殊/模糊指令、agent 场景。Pico 原来的 tool registry + schema 校验可以作为基础，但如果要变得新，应该引入 stateful tool environment、tool-use failure taxonomy、tool latency/cost accounting、tool-call replay 和 tool-result verifier。[R3]

MemoryAgentBench、InfiAgent、PaperArena 等工作说明，memory 不再只是“把历史总结一下”。新研究更关注增量多轮信息处理、外部化状态、长任务中 bounded context 的稳定性、工具使用效率。Pico 原来的 layered memory 是合理起点，但应该升级成：workspace state snapshot + recent action window + structured task state，而不是继续往 prompt 里塞摘要。[R2]

### 1.3 Coding Agent 评测正在从 SWE-bench 静态题集扩展到 live、long-horizon、multi-language、PR 级真实数据

SWE-bench 仍然重要，但现在只会说 SWE-bench 已经不够了。SWE-bench-Live 关注新近 GitHub issue，强调 live-updatable、Docker image、自动化 curation、减少数据污染；SWE-Bench Pro 强调企业级、长周期、多文件、大修改任务；SWE-Bench++ 则尝试从开源 PR 自动生成多语言仓库级任务。[R4]

这直接影响你的项目评测设计。Pico 原始 benchmark 如果只有几个 fixture repo 和固定 verifier，很容易显得像玩具。新的方向应该是：从真实仓库 commit / issue / PR 中自动构造任务，生成可执行环境，提取 test oracle，并记录 agent trajectory。即使你规模做不到上千，也应该做一个小而真实的 RepoTaskBench：比如 20 个仓库、80 个任务、覆盖 bugfix、test repair、refactor、dependency upgrade、CLI behavior fix、文档同步、前后端类型修复。

AIDev 和 failed agentic PR 研究也值得参考。它们把真实 GitHub 中由 Codex、Devin、Copilot、Cursor、Claude Code 等 agent 产生的 PR 作为研究对象，分析 merge 结果、CI 状态、review 动态和失败原因。[R5] 这说明行业下一步关注的不是“agent 能不能生成 patch”，而是“agent 生成的变更能否通过真实项目的 review/CI/merge 流程”。你的项目可以吸收这个视角：不要只看 verifier pass，还要看 diff size、touch files、CI pass、reviewability、irrelevant edit ratio、rollback rate。

## 2. 开源项目趋势：不要只看 SWE-Agent，重点看终端 agent、协议生态、skills/hooks、runtime 化

GitHub 本身有 Trending、Topics、Search，但没有一个官方稳定的“最速 STAR 榜”作为严肃研究口径。可以用 GitHub Trending、topics/coding-agents、star-history、OSS Insight、第三方榜单、Hacker News、X/Twitter、知乎/掘金/博客作为弱信号。弱信号的用法不是抄它，而是看生态正在堆哪些抽象：CLI、MCP、skills、hooks、subagents、AGENTS.md、trace、container sandbox、evaluation。

### 2.1 终端 coding agent：OpenCode、Gemini CLI、Codex CLI、AtomCode

OpenCode 是目前很值得看的终端/IDE/desktop coding agent 项目，它的定位是 open source coding agent，而且 GitHub 星数非常高。它代表了一个方向：coding agent 不一定从 IDE plugin 开始，而是先在 terminal-native workflow 站住，再扩展到 IDE/desktop。[R6]

Gemini CLI 是 Google 推出的开源终端 AI agent。它的仓库结构里能看到 evals、integration tests、memory-tests、perf-tests、skills 相关目录，以及大量围绕 session、history、tool output、performance 的工程演进痕迹。这个项目的价值不只是“接了 Gemini”，而是它将 CLI agent 当成长期维护的软件产品来做。[R6]

OpenAI Codex CLI 仓库也值得看。它是本地运行的 coding agent，仓库里有 AGENTS.md、codex-rs、codex-cli、SDK、hermetic tests、trace/latency 相关改动等工程痕迹。你可以学习它如何将 CLI、本地执行、patch、日志、测试和模型调用组织成产品，而不是学习它的模型。[R6]

AtomCode 是你提到的项目之一，定位是 Claude Code / Cursor Agent 的开源平替，Rust 实现，强调终端中自主读代码、改文件、跑测试、自我验证。它宣传“100% AI-generated codebase”这个点可以作为社交传播，但作为简历项目不建议跟。你可以参考它的 Rust terminal agent 结构和自我验证流程，但不要把“AI 写了项目”当亮点。[R6]

### 2.2 Personal agent/workstation：QwenPaw、Qwen-Agent、OpenClaw

QwenPaw 的定位不是纯 coding agent，而是 Qwen Personal Agent Workstation，偏个人助手/工作站，易安装、可部署到本机或云端、能接多个聊天应用并扩展能力。它对你的启发是：Agent runtime 不一定只服务代码，也可以有 workspace、channel、capability、app integration 的抽象。[R7]

Qwen-Agent 是 QwenLM 下的 agent framework，覆盖 instruction following、tool usage、planning、memory，并带 browser assistant、code interpreter、custom assistant 等示例应用。它适合参考模型-工具-应用的分层，以及 Qwen 生态如何组织 agent 能力。[R7]

OpenClaw 这种通用自动化 agent 的热度也说明，生态正在从“代码助手”扩展到“本地/云端可执行工作流”。但 OpenClaw 这类项目对你更大的启发不是安全故事，而是 plugin marketplace、skills、gateway、local control、workflow automation。也就是说，Agent Runtime 的价值会出现在“工具生态 + 执行环境 + 生命周期管理”上。[R8]

### 2.3 Harness/技能生态：OpenHarness、awesome-claude-code、Codex skills、Playwright MCP

OpenHarness 非常值得看，因为它直接把“Agent Harness”当显式项目来做，功能点包括 streaming tool-call cycle、API retry、parallel tool execution、token counting、43 tools、MCP、on-demand skill loading、skills/hooks/agents plugin ecosystem、context compression、persistent memory、session resume、permission modes、path/command rules、hooks、subagent spawning 等。[R7]

这类项目说明：如果你继续在 Pico 上只做“read_file/search/run_shell/write_file”，会显得太薄。更值得做的是 capability surface：工具动态发现、技能按需加载、hook 生命周期、subagent 管理、parallel tool execution、token/cost tracking、run trace 可视化。

Microsoft Playwright MCP 也值得看。它通过 MCP 给 LLM 提供浏览器自动化能力，并强调用结构化 accessibility snapshots，而不是依赖截图或视觉模型。这说明 MCP 不是“又一种工具调用格式”，而是将复杂环境抽象成 agent 可操作工具面的方式。[R9]

### 2.4 协议层：MCP、A2A、ACP

MCP 已经成为事实上的 tool/context integration 标准之一。官方规范将它定义为连接 LLM applications 与 external data sources/tools 的开放协议；Anthropic 的介绍也明确说 MCP 用于连接 content repositories、business tools、development environments。[R9]

A2A 关注 agent-to-agent interoperability，它解决的是不同组织、不同平台 agent 如何协作。ACP 则更偏 IDE/编辑器侧，Zed 将 Agent Client Protocol 定义为让任意 agent 接入编辑环境的开放标准，Gemini CLI 已经有 ACP mode / Zed 集成相关实践。[R9]

对你的项目来说，MCP/A2A/ACP 不一定都要实现，但至少要在设计报告里分清：MCP 是 agent-tool/server 协议，A2A 是 agent-agent 协议，ACP 是 agent-client/editor 协议。你可以先实现 MCP client + local tool manifest，再预留 ACP adapter。这样比“我接了 MCP”更有架构意识。

## 3. 为什么“安全审批”不适合作为主卖点：大厂关心的是 runtime，而不是弹窗

你说得对：如果项目主线是“高风险动作让人介入审批”，大厂实习不会觉得有多强。因为这类审批是最直接、最保守、最没有系统含量的降级策略。真正的大厂 Agent Runtime 岗位会关心的是：

1. 如何支撑海量 Agent 并发运行；
2. 如何管理容器生命周期、workspace 生命周期、依赖缓存和冷启动；
3. 如何做资源精细调度，包括 CPU、内存、磁盘、网络、GPU/推理资源；
4. 如何统一支持多硬件平台和多模型后端；
5. 如何让模型调用链路、上下文处理、工具执行、状态恢复形成可观测闭环；
6. 如何通过 benchmark 和线上指标持续优化 Agent 能力。

Google Cloud 的 Agent Sandbox / GKE 方向已经把 agentic AI 放到 Kubernetes 和 runtime isolation 语境下讨论，并提到 gVisor、Kata Containers 之类隔离机制。[R10] 腾讯云和阿里云的 Serverless AI Runtime 文章也都在强调会话亲和性、弹性伸缩、强隔离、有状态上下文、多租户等 agent runtime 需求。[R10] 这些和你贴的 JD 完全一致：不是“让用户确认危险命令”，而是做一个可以承载大量 agent 任务的运行底座。

因此你在项目里可以保留 policy，但不要把它写成“人工审批”。更好的表达是：

* tool action admission control：工具动作进入执行队列前的 admission；
* execution sandbox：每个 run 的容器/微虚拟机/进程隔离；
* workspace overlay：每个任务使用 git worktree 或 overlayfs，执行后产出 patch；
* resource quota：每个 run 的 CPU/memory/disk/network/time limit；
* dependency cache：依赖层、Docker layer、venv、node_modules 缓存；
* lifecycle state machine：Pending / Preparing / Running / Verifying / Completed / Failed / Recycled；
* scheduling policy：FIFO、priority、cost-aware、warm-pool-aware；
* trace and metrics：p50/p95 cold start、tool latency、test runtime、queue wait、tokens、cost、pass rate。

这样项目就从“工具安全”升级成“Agent execution platform”。

## 4. 建议项目重构方案：RepoPilot Runtime

### 4.1 总体架构

RepoPilot Runtime 建议拆成五个平面：

**第一，Control Plane。**
负责接收用户任务，创建 session/task/run，做任务状态机，调度 agent loop，维护 checkpoint，管理 run lifecycle。Pico 原来的 Pico.ask() 可以拆开：TaskAdmission、RunController、LoopExecutor、StopPolicy、ReportBuilder。不要再让一个 ask() 函数承担所有职责。

**第二，Execution Plane。**
负责 workspace 准备、容器准备、工具执行、命令执行、测试运行、patch 提取。这里是对 DeepSeek JD 最有价值的部分。你可以实现轻量版本：本地先用 Docker 或 subprocess + temporary worktree；高级版本再支持容器池、warm workspace、resource limit、network off、artifact collection。

**第三，Context Plane。**
负责 repo map、symbol index、file summary、diff summary、test map、token budget、context compaction、memory retrieval、prompt cache key。Pico 原来是 char-based section budget；你应该升级成 token-aware，并记录每一轮 context build 的 token usage、section pressure、compaction reason、被裁内容类型。

**第四，Model Plane。**
负责多 provider 调用、retry、streaming、structured output parsing、prompt cache、model routing、cost accounting。不要只写“支持 OpenAI/Claude/Ollama”。应该加入路由策略：低风险调查用 cheap model，patch synthesis 用 strong model，test failure diagnosis 用 reasoning model，summary/compaction 用 small model。指标记录每类模型调用的 cost、latency、success contribution。

**第五，Evaluation Plane。**
负责 benchmark task schema、fixture/workspace 构造、verifier、trajectory aggregation、ablation runner、dashboard/report。Pico 原来的 fixed benchmark 可以保留，但要升级成多维评测：pass rate、time to first patch、tool steps、test runs、irrelevant edits、container cold start、queue wait、context compression、cache hit、cost per solved task。

### 4.2 核心模块设计

**RunController。**
将一次任务拆成状态机：Created、WorkspacePreparing、ContextBuilding、ModelThinking、ToolExecuting、Verifying、Compacting、Checkpointing、Finished。每个状态都写 trace event。这样你在面试里能解释“Agent Runtime 如何观测和控制长任务”，而不是只说“循环调用模型”。

**WorkspaceManager。**
支持三种 workspace strategy：copy、git worktree、overlay。copy 最简单但慢；git worktree 适合多任务并发；overlay 适合容器运行和快速回滚。指标包括 workspace_prepare_ms、workspace_size_mb、changed_files、patch_extract_success、cleanup_success。

**SandboxWorker。**
每个工具执行请求不是直接 subprocess.run，而是生成 ToolJob，进入 worker。worker 带资源限制和工作目录绑定。轻量实现可以先本地进程池；进阶实现用 Docker container pool。指标包括 cold_start_ms、warm_reuse_rate、command_exit_code、stdout_tokens、stderr_tokens、affected_paths、timeout_count。

**ContextEngine v2。**
每轮 prompt 不再只是 prefix/memory/history/current request。建议拆成：System Contract、Repo Map、Task State、Relevant Files、Recent Observations、Open Questions、Candidate Plan、Tool Feedback、Current Request。所有 section 有 token budget 和 priority。compaction 不是静默裁剪，而是生成 ContextCompactionEvent：section、before_tokens、after_tokens、strategy、semantic_loss_risk。

**RepoMapBuilder。**
不要整仓索引到 prompt，但要生成轻量 repo map：目录树前两层、语言分布、配置文件、测试目录、入口文件、最近变更、符号索引摘要。对 Python/Go/TS 可以先做静态 parser：函数名、类名、导出符号、测试函数、路由/CLI entry。这个模块比“多读几个 README”更像 coding agent 的上下文工程。

**VerifierRouter。**
根据仓库类型选择验证命令：pytest、go test、npm test、cargo test、mvn test。支持 quick verifier 和 full verifier。quick verifier 用于每轮 patch 后快速反馈，full verifier 用于最终验收。指标包括 verifier_selection_accuracy、test_runtime_ms、fail_to_diagnosis_steps。

**MCPToolRegistry。**
实现 MCP client 不是为了炫协议，而是为了解决 tool ecosystem 动态接入。每个 MCP server 导入后生成 capability manifest：tools、resources、prompts、auth、side_effect、latency_estimate、trust_level。模型看到的不是全量 MCP 描述，而是经过 context budget 压缩后的 tool surface。这样才是 harness 设计，而不是“我连上了 MCP”。

**SkillLoader / HookEngine。**
借鉴 Claude Code / OpenHarness 生态。Skill 是按需加载的工作流说明，不常驻 prompt；Hook 是生命周期事件，如 before_tool、after_patch、before_verify、after_failure。你可以做三个技能：debug_test_failure、add_unit_test、refactor_small_function。Hook 做两个：after_patch_run_formatter、after_failure_collect_diagnostics。不要做太多，能评测收益即可。

## 5. 评测方案：不要再用“固定 6 个任务 + 通过率”作为核心证明

### 5.1 RepoTaskBench-Lite

构建 60-100 个任务足够用于实习项目。任务来源建议：

* 10 个 Python 小仓库：pytest、CLI、配置、依赖问题；
* 5 个 Go 小仓库：go test、接口修改、错误处理、并发 bug；
* 5 个 TS/Node 小仓库：类型错误、API client、测试修复；
* 每个仓库 3-5 个任务；
* 每个任务有 issue prompt、初始 commit、verifier、expected patch constraints。

任务类型不要只做 bugfix：

1. test failure diagnosis；
2. small feature；
3. dependency/config upgrade；
4. refactor with behavior preserved；
5. documentation-code consistency；
6. CLI behavior change；
7. flaky test repair；
8. type/interface repair。

每条任务都记录：任务复杂度、预期触达文件数、是否需要运行测试、是否需要搜索代码、是否需要读配置、是否需要跨文件修改。

### 5.2 指标体系

Agent 成功指标：

* resolve_rate；
* verifier_pass_rate；
* pass@budget；
* average_tool_steps；
* average_model_calls；
* average_test_runs；
* cost_per_resolved_task；
* time_to_first_valid_patch；
* time_to_final_pass。

Harness 效率指标：

* context_tokens_per_step；
* compaction_count；
* cache_hit_rate；
* repo_map_hit_rate；
* repeated_file_read_count；
* irrelevant_file_read_count；
* irrelevant_edit_ratio；
* tool_result_reuse_rate。

Runtime 指标：

* workspace_prepare_ms；
* sandbox_cold_start_ms；
* sandbox_warm_start_ms；
* queue_wait_ms；
* p95_tool_latency；
* p95_verify_latency；
* max_concurrent_runs；
* sandbox_reuse_rate；
* cleanup_failure_count。

恢复与鲁棒性指标：

* resume_success_rate；
* checkpoint_staleness_detect_rate；
* retry_recovery_rate；
* parse_error_recovery_rate；
* flaky_verifier_recovery_rate；
* partial_patch_rollback_success。

### 5.3 Ablation 设计

必须做 ablation，否则像自嗨。建议至少做 6 组：

1. baseline Pico loop；
2. * token-aware context；
3. * repo map；
4. * workspace manager / sandbox worker；
5. * skill/hooks；
6. * model routing；
7. * parallel patch candidates + verifier selection。

每组跑同一批任务，输出表格：

* resolve_rate；
* avg_cost；
* avg_latency；
* avg_tool_steps；
* context_tokens；
* test_runs；
* failure_category distribution。

这样你能回答“到底哪个模块有用”。这比原来写“我做了上下文、记忆、安全、评测”强很多。

## 6. 简历表述建议

不要写：

> Pico：本地代码智能体 Harness，支持工具调用、上下文管理、记忆、checkpoint、评测闭环。

建议写：

> RepoPilot Runtime：面向多仓库 Vibe Coding 的容器化 Agent Harness
> 设计并实现本地 Agent Runtime，将用户任务抽象为 session / task / run / tool job 四层状态模型，拆分 Control Plane、Execution Plane、Context Plane、Model Plane 和 Evaluation Plane；支持 git worktree workspace 管理、容器化工具执行、token-aware context compaction、repo map 检索、多模型路由、MCP tool registry、skill/hook 扩展和可复现实验评测。
> 构建 RepoTaskBench-Lite，覆盖 Python / Go / TypeScript 多仓库任务，统计 resolve rate、verifier pass、tool steps、context tokens、sandbox cold start、queue wait、cost per solved task 等指标；通过 ablation 分析 repo map、context compaction、model routing、sandbox warm pool 对任务成功率、延迟和成本的影响。

如果需要更偏算法/模型链路：

> 针对 coding agent 长链路任务中的上下文膨胀、工具反馈遗忘、模型调用成本高和验证闭环弱等问题，设计 token-aware context engine 和 model routing 策略，将 repo map、recent observations、task state、tool feedback 分层组织，并通过 prompt cache key、compaction event、failure taxonomy 和 verifier feedback 形成闭环优化；在多仓库任务集上对比不同 context policy 和模型路由策略的成本/效果。

如果需要更偏平台/runtime：

> 实现面向 Agent 的轻量容器执行平台：将工具调用下沉为 ToolJob，支持 workspace prepare、sandbox allocation、command execution、artifact collection、patch extraction、verification 和 cleanup 的生命周期管理；实现 cold/warm worker 对比、资源限额、并发调度和 trace metrics，评估不同 workspace strategy 与 sandbox reuse 策略对 p95 latency、throughput 和任务成功率的影响。

## 7. 不建议做的方向

第一，不要主打“高风险工具审批”。审批是最弱的安全策略。你可以保留 policy，但主线要变成 execution admission、resource isolation、workspace lifecycle、tool job scheduler。

第二，不要只接新模型。Qwen、DeepSeek、Gemini、Claude、OpenAI provider 都可以接，但 provider 数量不是深度。深度在 model routing、prompt cache、latency/cost accounting、fallback、structured output reliability。

第三，不要只接 MCP。MCP 现在很热，但“接上 MCP server”很容易。真正值得做的是 MCP capability manifest、tool surface compression、per-server trust level、tool-call replay、MCP tool latency/cost/side-effect 观测。

第四，不要做大而空的 multi-agent。子 agent 不是越多越好。先做 read-only investigator、patch proposer、verifier 三类角色，并定义消息协议、产物格式、冲突处理和成本上限。

第五，不要继续沿用 Pico 原始数字。所有实验数字必须自己跑。即使数字不漂亮，也比背别人 PDF 的数字强。大厂面试最怕你讲不清实验构造、失败样例和指标含义。

## 11. 最后判断

如果你只是把 Pico 改成“更安全的 Pico”，价值不够。大厂 Agent 实习真正吃香的是：你能不能理解 Agent 不只是 prompt，不只是 tools，而是一个运行时系统。这个系统要解决任务接入、上下文构造、模型调用、工具执行、workspace 管理、容器生命周期、状态恢复、并发调度、成本控制、评测回归和产品体验。

Pico 已经有一个最小 agent loop，它适合被改造成 runtime baseline。你应该做的不是换皮，而是从“单机脚本式 coding harness”推进到“可调度、可观测、可评测的 Agent Runtime”。这样你在面试里讲的就不是“我做了一个本地代码智能体”，而是“我研究并实现了 coding agent 在真实仓库任务中稳定执行的系统底座”。
