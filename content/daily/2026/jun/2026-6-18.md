### stablepay
好烦人啊好烦人，老板要约个时间对一下，但是我们还没有拉多少人测试，大家都不肯私聊拉人！
以后发布到npm的话就这：
```bash
npm publish --access public
```
发布后从npm安装测试：
```bash
openclaw plugins install npm:@bubblevan/stablepay-agentpay-dev@0.3.19 --force
openclaw plugins inspect stablepay-agentpay-dev --runtime --json
```
仓库里已经有 Agent 味的底子：stablepay_doctor、stablepay_onboard、trace 文档、支付限额 guardrail 都在了。但入口仍绑定 OpenClaw，README.md:163和 openclaw.plugin.json:9是插件中心化；package.json 也还没有真正的 CLI bin；支付链路仍会从本地取 solanaRpcUrl 并构建交易，src/pay_settlement.ts:76、src/tx_builder.ts:63 正好说明 WSL/RPC 摩擦是真问题。

P0
1. gateway settlement mode：本地只签 intent/message，gateway 负责 RPC、blockhash、ATA、广播、查询。
2. StablePay CLI：doctor / onboard / demo / merchant buy / pay --json，先让任何能跑 shell 的 Agent 都能用。
3. Runtime trace：把现在 eval trace 扩展到真实运行链路，但注意不要存私钥、完整签名、完整 signed tx，也不要把模型隐藏思考当成可依赖数据。
4. 产品化错误报告：所有失败都返回“卡在哪一步 + 下一条命令”。

P1
1. MCP server，但只暴露高层工具，不要把 19 个 OpenClaw primitive 原样搬过去。
2. stablepay install codex / claude-code / openclaw / hermes。
3. Web Playground 或飞书机器人 Demo，让用户 1 分钟内看到支付价值。
4. doctor --fix，但必须做成显式修复计划，别让一个命令静默改钱包/密钥/支付状态。

P2
1. 远程 MCP、StablePay Link、ChatGPT App、商家 marketplace demo。
2. 这里我觉得 StablePay Link 可能可以提前，因为它不依赖任何插件生态，传播成本很低。

第一，trace 不是“把所有思考过程都存下来”。真实产品里应存 tool call、参数摘要、结果摘要、耗时、token、错误、状态迁移、trace/span id。支付类产品尤其要做脱敏和 hash 化。

第二，MCP 是对的，但用户心智不该是 MCP。官方资料也支持这个方向：Codex 当前支持 MCP STDIO 和 Streamable HTTP，Claude Code 文档也把 HTTP MCP 作为远程云服务的推荐传输方式；OpenAI Apps SDK 也确实是 MCP server + ChatGPT UI 的形态。资料：OpenAI Codex manual、[Claude Code MCP docs](https://code.claude.com/docs/en/mcp)、[OpenAI Apps SDK](https://developers.openai.com/apps-sdk/)。
