#### [简历](https://chatgpt.com/c/699adfba-50fc-8331-a3b7-f29109552436)
先说结论：如果不投 Web3 岗，而是投 **AI Agent / AI 后端 / Agent 平台 / AI Native 工程** 这类岗位，你的项目要重点往这几个方向去贴： 

* **Agent 系统能力**：Agent workflow、工具调用、支付决策策略、评测与优化
* **后端工程能力**：Go、微服务、网关、缓存、异步任务、监控、可用性
* **知识/检索能力**：RAG、知识库、提示词优化、知识注入
* **系统化落地能力**：高并发、高可用、幂等、可观测、部署上线

这和现在公开 JD 的方向是对得上的：
字节的 Agent/搜索 Agent 岗会明确要 **LLM Agent、RAG、Prompt Engineering、端到端系统、对齐/评测**；小红书会明确提 **DeepResearch Agent、PlanExecutor、Multi-Agent、评估体系、稳定可靠的运行方式**；偏 AI 后端的岗位又会明确要 **Go/Java/Python、分布式、高并发、微服务、DB、中间件、网关、监控**；腾讯这类后端岗也会强调 **把 LLM 调度能力集成进可扩展架构并做性能优化**。

这样就会非常贴现在 Agent 岗的要求。

##### ChatGPT 提醒
以后写简历时，最容易犯的错是：
* **错法**：把一堆技术名词全堆上去
* **正确写法**：每个技术点都绑定一个“你解决的问题”

比如：

* 用 Redis，不是因为会 Redis
  → 而是因为你要做 **支付幂等 / 会话状态 / 缓存查询**
* 用 RocketMQ，不是因为会 MQ
  → 而是因为你要做 **异步状态更新 / 审计追踪 / 最终一致性**
* 用 RAG，不是因为面试喜欢
  → 而是因为你要让 Agent **理解 skill 元信息 / 支付条款 / 价格约束**
* 做 Agent workflow，不是因为概念新
  → 而是因为你要让 Agent **在自动买 / 要确认 / 拒绝之间做可解释决策**

需要后面这套 **“面试问答版”**，比如面试官问你：
* 为什么用 Go / CloudWeGo？
* 6 个微服务怎么拆？
* Agent 技术栈在这个项目里到底体现在哪？
* 你不是投 Web3，为什么做这个项目也能匹配 Agent 岗？


#### 专利论文技术产出
这件事现在就该开始想，不是因为你们一定要真的投论文/申请专利，而是因为**“可专利/可论文化”本质上是在逼你把项目做成一个有方法论、有边界、有评测的技术方案**。这对创赛、对答辩、对简历都很有帮助。

你们现在项目里已经有几个天然的技术抓手：
一期明确是在 OpenClaw/ClawHub 里做“付费 skill 购买闭环”，核心机制包括 **HTTP 402 支付挑战、W3C DID 身份、自动购买阈值/单次限额、购买验证、防绕过校验**，而且是通过“在 skill 中插入支付相关内容 + 用户侧官方 skill + 平台后端服务”完成的。
后面的讨论又把项目往更大的方向拉：**不只是卖单个 skill，而是让 AI Agent 获得“自主支付/自主购买”的能力，并且未来可以泛化到更广的商品形态**；同时你们也意识到 “skills 太小、商品市场未形成、需要找到中间态垂类或包装层”。 
所以建议不要把“专利/论文点”想成“很高深的新算法”，而是分三层来想：

##### 一、先分清：什么更适合写专利，什么更适合写论文
| 对比维度       | 更适合写专利的特征                                                                 | 更适合写论文的特征                                                                 |
|----------------|-------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| 核心特征       | 有**明确流程**、系统模块、输入输出；<br>能说清“以前怎么做、现在怎么做”；<br>强调**工程机制、系统组合、流程创新**；<br>无需强数学创新，但需有“具体实现路径” | 有一个**明确的研究问题**；<br>有清晰的 baseline/ablation/量化指标；<br>实验可复现；<br>强调“为什么这个方法更优” |
| 项目适配性     | 项目内此类内容占比高（如支付策略、验证机制、封装层等方向均适配）                     | 项目内有适配内容，但更适合写成**系统论文/方法论文/benchmark风格小论文**，而非纯理论论文 |
| 写作核心重点   | 聚焦“怎么做”——落地路径、模块设计、流程创新点                                         | 聚焦“为什么优”——问题拆解、实验对比、指标验证方法优越性                               |

##### 二、最值得你们优先考虑的 8 个方案点
按“**最容易做实 → 最容易讲清楚 → 最容易写成东西**”排序。
| 方向编号 | 一句话总结 | 为什么适合 | 更像专利的写法 | 更像论文的写法 | 难度 | 核心推荐/补充说明 |

### 本科毕设
这篇发表于ICRA 2025的VLM-Social-Nav围绕**人机共享的社交导航场景**，采用**客观量化指标+主观评价指标**结合的方式界定真实世界实验指标，且所有指标均基于**Turtlebot实体机器人**在4个真实世界社交导航场景中的实测结果，对比DWA、BC等传统导航基线方法，确保指标的落地性和对比有效性，具体界定如下：
1. **核心客观量化指标**
    - **平均成功率**：定义为机器人在社交场景中完成导航任务（到达目标位置）的比例，论文中该指标实现了**至少27.38%的提升**，是衡量导航基本性能的核心指标；
    - **平均碰撞率**：定义为机器人在导航过程中与人类、环境物体发生碰撞的概率，论文中该指标实现了**19.05%的改善**（即碰撞概率显著降低），聚焦社交场景中机器人的运动安全性。
2. **社交合规性主观评价指标**
    设计**用户研究评分**作为核心主观指标，衡量机器人导航行为是否符合人类的社交期望（如保持社交距离、礼让行人、避免干扰人类活动等），这是社交导航区别于传统导航的关键指标，论文通过该指标验证了其方法生成的行为是**最符合人类社交认知**的。

此外，论文将指标与模型设计深度绑定：通过VLM评分模块将**人类主观的社交合规要求**转化为**可计算的成本项**，让隐性的社交规则（如“不要从两个人中间穿过”）量化为底层规划器能识别的数值约束，确保客观指标能直接反映社交合规性，实现了“指标-模型-实体机器人动作”的闭环。

**开放词汇的不确定性**是VLN真实世界实验的核心痛点，再结合社交导航的特性，整体难点可归纳为5点，也是当前VLN从仿真走向真实的主要障碍：
1. **自然语言指令的开放词汇/模糊性**
    真实世界的导航指令由人类自然表述，无固定范式，地标（如“桌子”vs“圆桌”）、方向（如“往前走一点”vs“走5米”）、动作（如“绕过去”vs“从旁边过”）的描述高度开放，缺乏统一的视觉/空间锚点，导致VLM难以精准将开放词汇与真实环境的视觉信息对齐，且不同用户的表述习惯差异会进一步放大这种不确定性。
2. **社交导航的隐性规则难以量化**
    传统VLN仅需衡量“是否到达目标”，但**社交VLN**需要兼顾人类的社交期望，而这些期望是隐性的（如不同场景的社交距离、行人礼让的优先级），无统一的量化标准，仅靠客观指标（成功率、碰撞率）无法完整衡量导航性能，必须引入主观评价，增加了实验设计的复杂度。
3. **真实环境的动态性与不可复现性**
    真实世界的社交场景中，人类的行为是随机、动态的（如行人突然转身、停下），社交实体的位置和动作无固定规律，导致实验的可复现性差，难以像仿真环境那样控制变量，指标的统计结果易受环境动态变化影响。
4. **感知与执行的现实噪声**
    实体机器人的传感器（摄像头、激光雷达）存在精度误差，且真实环境的光照变化、物体遮挡、纹理模糊会影响VLM的视觉理解，导致“开放词汇-视觉实体”的匹配失败；同时，机器人的运动控制误差会让规划的动作无法精准执行，进一步放大实验误差。
5. **仿真到真实的领域差距**
    现有VLN模型多在仿真环境中训练（固定的物体纹理、布局、指令范式），而真实世界的环境复杂度、指令开放性远高于仿真，模型易出现“仿真表现好，真实表现差”的问题，实验设计需要额外考虑模型的泛化能力验证。

针对上述难点，尤其是**开放词汇不确定性**和**社交规则量化难**，论文提出了轻量化且高效的解决方案，让VLN在真实社交场景中落地：
1. **感知模型+VLM的双层架构，弱化开放词汇的歧义**
    先用专用感知模型检测真实环境中的**核心社交实体**（人类、关键家具/地标），将开放的视觉信息抽象为结构化的实体标签，再将这些标签作为提示词输入VLM，让VLM无需处理无边界的开放视觉信息，仅专注于社交规则的理解和推理，减少了“开放词汇-视觉信息”对齐的难度。
2. **VLM评分模块实现社交规则的量化**
    让VLM基于人类社交认知为机器人的候选动作打分，生成**社交合规成本项**，将模糊的社交期望（如“保持1米社交距离”）转化为规划器能识别的数值约束，让社交合规性可被客观指标间接反映，解决了隐性规则量化的问题。
3. **减少对大训练数据集的依赖，提升对开放指令的适应性**
    传统VLN模型依赖大规模标注的“指令-视觉-动作”数据集，易被固定的指令范式束缚，而VLM-Social-Nav直接利用VLM的通用世界知识（如人类的社交习惯），无需针对特定场景/指令微调，能更好地应对真实世界的开放词汇指令。
4. **客观+主观指标的互补设计，覆盖导航性能与社交合规性**
    用成功率、碰撞率保证导航的**基本功能性**，用用户研究评分衡量**社交合规性**，既避免了纯客观指标无法反映社交需求的问题，也避免了纯主观指标缺乏量化依据的问题，让实验指标能完整评价真实世界的社交VLN性能。
