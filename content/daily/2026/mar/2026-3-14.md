#### 阿里云 API 网关
| 层面 | 技术选型 | 说明 |
| --- | --- | --- |
| 自建 API Gateway 微服务 | 用 Hertz 框架自研 | 你负责的这个服务，做业务逻辑路由 |
| 阿里云 API 网关（云产品） | 阿里云托管服务 | 接入层，做安全防护、限流、协议转换 |

根据 tech.md 第 93-95 行：
API 网关 | 阿里云 API 网关 | - | 统一接入、认证鉴权、限流保护

以及架构图：
```
┌─────────────────────────────────────────────────────────┐
│ 阿里云 API 网关（路由、鉴权、限流、协议转换）            │
└─────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────┐
│ 业务服务层 (你的 API Gateway 微服务 + Payment Service 等) │
└─────────────────────────────────────────────────────────┘
```

为什么要买阿里云 API 网关？

| 阶段 | 是否需要 | 原因 |
| --- | --- | --- |
| 开发阶段 | ❌ 不需要 | 本地开发可以直接访问你的自建 API Gateway |
| 内部测试 | ❌ 不需要 | 团队内部测试可以用自建网关 + 内网穿透 |
| MVP 上线 | ⚠️ 建议有 | 生产环境需要 DDoS 防护、SSL 证书管理、限流保护 |
| 正式对外 | ✅ 必须 | 对外暴露域名时需要专业网关的安全能力 |

老板提出购买的原因可能是：

1. 对外暴露域名 - 让开发者可以通过 https://api.stablepay.co 访问
2. 安全防护 - DDoS 防护、WAF、防刷机制
3. 合规要求 - 跨境支付服务需要满足一定安全标准
4. 运维便利性 - 阿里云网关提供监控、日志、限流等开箱即用

如果项目进度：
- 还在开发阶段（1-2 个月内不上线）→ 暂时不需要购买，先用自建网关开发
- 准备 MVP 验证（近期要对外）→ 可以购买基础版，费用不高（约几百元/月）
- 正式商业化 → 必须购买，且需要高配版本

替代方案：
- 初期可以用 Nginx + 自建网关 作为临时方案
- 配合 Cloudflare 免费版做 CDN 和基础 DDoS 防护
- 等业务验证后再迁移到阿里云 API 网关

| 问题 | 建议 |
| --- | --- |
| 两个服务能做什么？ | 接口骨架 + 本地 demo，无法完成真实支付 |
| 下一步优先补充？ | Blockchain Adapter（链上交易）和 DID Service（身份） |
| 现在买阿里云网关？ | 如果 1-2 个月内不上线，可以暂缓；要对外暴露则建议购买 |

- inference 不计算 eval 平均指标（不像 --run-type eval 那样输出 success/spl 均值）而是把每个 episode 的逐步轨迹信息（动作执行后的状态/位姿等）写到 PREDICTIONS_FILE
- 常用于提交结果、回放分析、或后处理
- 简单说：eval 看“分数”，inference 导出“轨迹/预测明细”

理论上有两类agent：
- nonlearning_havlnce.yaml：非学习基线（RandomAgent / HandcraftedAgent）
  - `HandcraftedAgent` 会先随机右转若干次，再前进 37 步，然后停。是一个“手工规则基线”

#### GroundingDINO
原意是主要给 `HUMAN_COUNTING` 用的，所以关了就好，但是`BaseVLNCETrainer` 里会**无条件**导入并初始化 `Detector()`（也就是 GroundingDINO），即使 `HUMAN_COUNTING=False`。
