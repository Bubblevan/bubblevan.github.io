如果 OpenClaw 真的很好的话，为什么还要做 NemoClaw 呢？老黄英明，是负责任的工程师，不像某些网红，只管推荐不谈风险不保证安全，本身design，code，validate，refine的流程还是分拆成不同agent或者人主动check in，用opus 4.6或者chatgpt 5.4都差不多，核心还是要架构梳理清楚

Agent Eng 的核心假设是：在 合适的 ctx 下 Agent 一定能做对事情
所以Reviewer 也好，还是 Harness 也好，本身其实是 为了提供更好的 ctx。
然而很多时候 TDD 已经能够提供 足够的 ctx 了。Reviewer 的意义其实是在一个更干净的上下文下再来审视一下 是否和用户 的 origin intent 匹配，因为在 负责 实现的 agent 的 ctx 中已经充斥了太多 实现噪声了。
所以其实 也可以反过来 把这个事情做成一个 recursive
一个 high level planning agent fork 一个 subagent 做实现，subagent 把实现结果如何验收回报给 planning agent。
这样就实现了 recursive 一层再回来，通过 这样的方式隔离的 ctx stack。
而现在大部分人做的 一个 agent plan 一个 agent code ，一个 agent review ，实际上是把 upper level 的 ctx 拆分到了 plan 和 review 两个 agent 里，我认为是 不完美的做法。
