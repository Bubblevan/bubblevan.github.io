#### 进度
**1. 现在 A1 / A2 到底差什么**
如果按“能不能演示主要业务闭环”来讲：

- `A1` 现在是“**降级版可跑，正式版未完成**”。
- `A2` 现在是“**微服务主链已跑通，正式版和网关签名版还没完全收口**”。

更详细地说：

`A1` 现在已经具备的部分是：
- `API Gateway -> DID Service` 的创建 DID / 钱包地址流程是通的，你已经实测过 [main.go](d:/MyLab/StablePay/payment-service/cmd/payment-service/main.go) 之外的 DID 链路。
- 降级联调里，我们已经能用 Query 的内部同步接口把“奖励到账”这件事 mock 出来，然后在 Query 里看到余额变化。

但 `A1` 距离 PRD 真正完成，至少还差这些：
- 缺少真正的 `X 验证页 + 后端验证接口 + tweet_url 校验 + DID 与 X 账号绑定持久化` 这一整段业务实现。现在仓库里 `verification-service` 负责的是“购买验证”，不是“X/Twitter 验证奖励”。你能看到 [handler.go](d:/MyLab/StablePay/verification-service/handler.go) 和 [http_server.go](d:/MyLab/StablePay/verification-service/http_server.go) 里都是 purchase verify，而不是社交验证。
- 缺少“验证成功后发 1 USDC 奖励”的真实资金发放链路。也就是说现在没有真正的 reward mint / reward transfer / reward tx 记录。
- 缺少“奖励到账后由 Query 真实体现”的正式写入链。我们现在用的是 mock sync，不是真实奖励服务发出来的事件。
- 如果以后要接真实 X，确实还差 HTTPS / SSL 这层，因为真实 OAuth / callback / 外部校验通常要求可信 HTTPS 地址。这一点不是伪需求，是真的会挡住正式联调。

所以 `A1` 不能只说“差 X 验证和 1 USDC 奖励”这么短。更完整的说法是：
- 它差的是 **整条“社交验证子链”**，包括前端入口、后端校验、绑定存储、奖励发放、奖励落账，以及真实环境下所需的 HTTPS 能力。

`A2` 现在已经具备的部分是：
- `payment-service` 直连调用可以成功创建支付，产生 `tx_id / tx_hash`。
- `payment_events` 会发到 RocketMQ。
- `verification-service` 能消费消息并写入购买记录。
- `query-service` 已经能把购买记录展示出来，`transactions.items` 和 `balance` 都是实测有数据的。
- 我已经实测打通了 `Payment -> RocketMQ -> Verification -> Query` 这条微服务主链。

但 `A2` 距离“正式可交付 / 完整产品态”还差这些：

- 差 **真实 DID 验签闭环**。  
  现在 `payment-service` 直连时，DID 验签其实是 stub，见 [client.go](d:/MyLab/StablePay/payment-service/internal/adapter/rpc/client.go)。`Validate(...)` 现在直接 `return true, nil`，所以直连 payment-service 的时候，并没有真正校验用户签名。
- 差 **真实链上转账与真实余额检查**。  
  同一个文件里 `ExecuteTransfer(...)` 现在返回的是 `simulated_tx_hash_*`，`CheckBalance(...)` 也直接假定余额足够，`QueryTxStatus(...)` 也直接回 confirmed。也就是说现在没有真实 Solana RPC、没有真实链上 USDC 转账、没有真实链上确认。
- 差 **真实用户私钥签名链**。  
  PRD 里用户侧支付应该是 skill/agent 用本地私钥对支付请求签名；现在我们在直连 payment-service 演示时，`signature` 字段实际上填什么都能过，因为 payment-service 的 DID validator 是 stub。
- 差 **gateway 版 DID 签名调用完全收口**。  
  `api-gateway` 本身是要求 DID 鉴权的，见 [config.yaml](d:/MyLab/StablePay/api-gateway/configs/config.yaml) 和 [auth.go](d:/MyLab/StablePay/api-gateway/internal/interfaces/http/middleware/auth.go)。我已经验证了 `did-service` 的真实签名校验是工作的，见 [did_app_service.go](d:/MyLab/StablePay/did-service/app/did_app_service.go)。但在当前运行态里，`POST /api/v1/pay` 通过 gateway 仍然报 `signature invalid`。这说明 A2 还差最后一层：**gateway canonical string、客户端签名构造、did-service verify-sig 三者之间还有一处细节没对齐**。
- 差 **真实价格/真实扣费策略**。  
  当前 `payment-service` 里的 `GetPaymentRequirement` 价格还是硬编码 `"0"`，见 [payment_service.go](d:/MyLab/StablePay/payment-service/internal/application/service/payment_service.go)。所以真正的“低价自动购买 / 高价二次确认 / 余额不足提示最少充值金额”这些 PRD 级体验，目前还没有真实价格源支撑。
- 差 **真实余额语义**。  
  现在 Query 的余额已经不是写死值了，但如果没有真实充值或真实奖励入账，购买后的余额会变成负数。这说明 Query 展示链已经通了，但“产品可接受的资金来源闭环”还没通。

所以一句更完整的总结是：

`A1` 目前是“创建 DID 与钱包链已通，社交验证奖励链未完成”；它缺的不是一个单点，而是整条 X 验证子链，包括验证页、tweet 校验、DID-X 绑定、1 USDC 奖励发放、奖励入账展示，以及真实环境下所需的 HTTPS 能力。  
`A2` 目前是“支付微服务主链与 Query 展示链已通，但正式支付链仍未完成”；它还缺真实 DID 验签、真实用户私钥签名、真实 Solana 转账与余额检查、真实价格与扣费策略，以及经由 `api-gateway:8080` 的签名契约最终对齐。换句话说，现在 A2 已经证明了 `Payment -> RocketMQ -> Verification -> Query` 的工程链路成立，但还不能说已经达到 PRD 所要求的真实支付产品态。
