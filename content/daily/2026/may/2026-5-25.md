#### AI Agent 开发生态的 2026 变局

##### 1.1 范式转移：从 Python 霸权到两强并立

2024 年，如果你问一个 AI 工程师用什么语言写 Agent，答案几乎是清一色的 Python。到了 2026 年，这个答案不再确定了。Mastra AI 创始人 Sam Bhagwat 在 2025 年 6 月透露的一个数据引爆了社区：**大约 60% - 70% 的 Y Combinator X25 批次 AI Agent 创业公司正在使用 TypeScript 构建**——这意味着在 AI 这个原本由 Python 统治的领域，TypeScript 已经在前沿创业公司中实现了"静默逆袭"。

与此同时，TypeScript 在 GitHub 上的增速（年增长 66%）已经超越了 Python。2026 年 2 月，一篇题为《I Rebuilt My Agent Framework From Scratch. Here's Why TypeScript Won》的技术博客在开发者社区刷屏——文章作者的核心判断是：TypeScript 在 2025 年正式在 GitHub 贡献者数量上超越了 Python，**全栈的 TS 开发者才是最庞大的生产级代码编写群体**，然而整个多 Agent 编排生态却是 Python-first 的。他认为这个缺口即将被迅速填补。

##### 1.2 为什么 TS 开发者希望 Agent 框架长在 TS 生态里

简单来说，一套技术栈如果能同时满足前端、后端、AI Agent 编排甚至桌面应用的开发需求，就能极大降低团队的维护成本和上下文切换开销。因此，当 TS 开发者想要构建多 Agent 系统时，他天然希望框架的语言、类型系统和工具链与他已有的 Web 开发栈完全统一。

#### Python 生态 vs. TypeScript 生态

| 维度 | Python 生态 | TypeScript 生态 |
|---|---|---|
| **核心优势** | AI/ML 生态最成熟，数据科学生态链（NumPy、pandas、PyTorch）无可替代，适合研究、复杂多 Agent 协作、数据密集型任务 | 类型安全 + 全栈统一语言，适合 Web 产品、Serverless、高并发生产环境 |
| **代表框架（2026）** | **LangGraph** (LangChain Inc., MIT, ~8.2k+ 但累计 47M+ PyPI 下载) —— 状态图 Agent 编排，被 Uber、LinkedIn、Klarna 等巨头用于生产环境<br>**CrewAI** (社区驱动, MIT, ~25.5k+ stars) —— 角色扮演式多 Agent<br>**AutoGen** (Microsoft Research, Apache 2.0/MIT, ~38.6k+ stars) —— 对话式多 Agent<br>**OpenAI Agents SDK** (MIT, ~15k+ stars) —— 轻量级 Agent 开发<br>**Google ADK** —— 多模态 Agent<br>**PydanticAI** (Pydantic 团队, MIT, ~8k+ stars) —— 类型安全 Python Agent<br>**LlamaIndex** (~37k+ stars) —— RAG + 数据 Agent | **Mastra** (MIT, ~22k+ stars) —— TS-first Agent 编排 SDK，内置 workflows、memory、MCP、observability，被 Replit、PayPal、SoftBank 等企业采用<br>**Vercel AI SDK** —— LLM 应用 TS 工具包，注重 streaming 和 bundle 优化 |
| **多 Agent 编排模式** | 角色扮演型、对话型、认知型 | 蓝图化工作流 Agent + 工具调用 + 动态路由 |
| **MCP (Model Context Protocol) 支持** | 2026 年所有框架都原生支持 MCP | 同样原生支持 MCP，而且和 TS 现有工具链集成更顺畅 |
| **部署环境** | 全平台通用，包括本地 GPU 环境 | 天然适配 Cloudflare Workers、Vercel、Deno Deploy、Edge.js 等现代 JS/TS 运行时 |
| **学习曲线** | 对于数据工程师友好 | 对于全栈 Web 开发者友好 |
| **2026 年最常见的生产方案** | Python 做计算层 | TypeScript 做编排层和用户端 |

2026 年最常见的生产方案是**混合架构：Python 做计算层 + TypeScript 做编排层和用户端**——用 Python 调用 PyTorch 等原生计算库，用 TS 完成 Agent 工作流编排和用户交互，两套语言各取其长。

#### "Agent Runtime / Web runtime"？

首先需要区分 AI 领域中 runtime 的两个不同层面。

##### 3.1 底层 AI Runtime / Agent Runtime

这个层面的 runtime 是 2026 年最受关注的基础设施概念。2026 年 Gartner 将其定义为："**一个专为安全托管和执行 Agent 化应用程序而设计的执行环境**"。

一个形象的比喻是：LLM 是引擎，runtime 是底盘、变速箱和方向盘——它将原始而混乱的计算动力转化为可控、可预测、可持久化的实际工作流。

2026 年 4 月，Cloudflare 在 Agents Week 上发布了 Project Think Agents SDK，主打**durable execution with crash recovery and checkpointing、sandboxed code execution、persistent sessions with tree-structured messages**。同时 OpenAI 也在同一天更新了其 Agents SDK，新增了**native sandbox execution 和 model-native harness**。这意味着无论是 Cloudflare、OpenAI 还是 Google，都将 runtime 定位为 Agent 能否落地的关键战场。

对于一个 AI Agent 来说，runtime 需要解决的核心问题是：

1. **持久化内存**：Agent 必须记住跨会话的状态，像一个靠谱的助手，而不是一条金鱼——runtime 负责管理这个记忆仓库；
2. **安全策略约束**：防止用户通过 prompt 注入等方式让 Agent 突破业务规则，runtime 负责在 LLM 执行前和执行中注入硬性安全边界；
3. **容错与断点续传**：如果 Agent 在步骤 4 崩溃，runtime 必须能从步骤 4 恢复执行，而不是整个任务失败；
4. **沙箱隔离**：不同 Agent 之间互相隔离，未授权代码不能接触系统核心。

##### 3.2 上层 Web Runtime / JS Runtime

这是聊天记录中更直接指向的层。对话中的"理想语言是 TS 语法 + Python 生态 + TS 的 Web runtime 整合在一起"，其实是在呼唤一个能够混合三者的执行环境：**用 TS 的 Web 运行时调用 Python 的原生计算库**。

**Edge.js** 是由 Wasmer 公司于 2026 年 3 月推出的一款开源 JS 运行时，它**利用 WebAssembly 沙箱技术**去安全地执行 Node.js 工作负载，目标正是 AI Agent、MCP 服务器和边缘计算。它的核心突破在于：可以在安全隔离的 WebAssembly 沙箱中运行原生的 Node.js 应用，不需要传统 Docker 容器，并能在云端、边缘甚至浏览器中执行。该 runtime 目前已兼容 V8 和 JavaScriptCore 两种引擎。

此前，WebAssembly（Wasm）已经从浏览器原生的"页面优化工具"，进化为 2026 年边缘端和服务器端的**一等公民**——轻量级、高性能、易扩展的 Wasm 运行时（如 CNCF 旗下的 WasmEdge）正越来越多地用于在边缘设备上直接运行大语言模型推理任务。

#### 综合判断
> "Python 需要一个良好的 Agent 框架和以它为核心的 General Agent 平台了，要不然 风头要被 JS / TS 抢完了"

1. **市场信号已然明确**：最前沿的初创公司（YC X25 批次）已经有超过六成在 TS/JS 生态构建 AI Agent；
2. **TS 拥有开发者基础优势**：GitHub 上 TS 贡献者总量已经超过 Python，全栈开发者规模庞大；
3. **TypeScript 正在向推理层渗透**：随着 WebAssembly、边缘计算和相关 runtime 的发展，TS/JS 已不再局限于前端和 API 层，而是开始进入原本 Python 主导的推理和 Agent 编排层；
4. **Python 生态的传统壁垒（PyTorch、数据科学生态）仍在**，但若长期缺乏一个与 TS/JS 对等的、轻量且易部署的通用 Agent 平台，可能在 AI Agent 的生产部署领域被 TS 生态部分替代。


#### “Harness”是什么？——给AI的“操作系统”

简单来说，**Harness是包裹在大语言模型（LLM）外的一整套确定性系统**，负责规划、控制、调度和兜底，将大模型不可控的“智力”转化为稳定可靠的生产力。

业内有个比喻：**模型就像一颗强大的CPU，而Harness就是让CPU能发挥作用的操作系统**。在2026年，生产级AI Agent的共识公式是：

> **生产级 Agent = (AI模型 + 能力组件) × Harness管控系统**

一个成熟的Harness系统，需要解决以下关键问题：

*   **任务调度与管理**：核心是规划和调度。
*   **记忆管理与上下文**：负责Agent的长期记忆和短期上下文。
*   **工具调用 (Tool Use)**：赋予Agent调用外部API或执行代码的能力。
*   **安全与合规**：设置规则，保障安全。

#### Harness vs. Framework

为了说清楚“超越一种 harness 框架”，就绕不开Harness和传统开发框架（Framework）的区别。

| 对比维度 | **Agent Harness (驾驭层)** | **Agent Framework (框架层)** |
| :--- | :--- | :--- |
| **比喻** | 整车 / 操作系统 | 零件 / 组件库 |
| **核心哲学** | **封装复杂性**，提供一个开箱即用的“成品” | **提供灵活性与控制力**，让开发者自由组装 |
| **产出目标** | 直接可运行、可交付的Agent应用 | 一个可在此基础上进行开发的底层系统 |
| **用户画像** | 追求交付速度，希望快速将AI能力投入生产的团队 | 追求深度定制，需要对执行过程进行精细控制的团队 |
| **代表项目** | OpenAI Deep Research、OpenClaw | LangChain, CrewAI, Mastra |

#### Harness下的任务调度与管理
群友认为Lark CLI能做到“超越”，原因在于它代表了更高层次的工作流抽象，而不仅仅是完成任务分解和分发。一个Harness下的任务调度管理会涉及：

*   **工作流 (Workflow) 编排**：将任务用流程图定义，确保执行的确定性，便于追踪。
*   **状态管理与断点续传**：持久化Agent执行状态，支持随时中断和恢复，对长时间任务至关重要。
*   **智能路由与资源分配**：根据任务类型，动态调度后端资源，保证运行效率。
*   **容错与可观测性**：内置重试和监控，能定位问题根源。

#### lark-cli：被群友认为“超越”的实践

**lark-cli**是飞书在2026年3月开源的一款官方命令行工具。它之所以被认为能做到“超越”，是因为它为AI Agent提供了一套**原子化的、高层次的“业务原语”**。一个复杂的任务，在“Harness”层面被拆解为几个步骤，最终落到Lark CLI这样的具体指令上：

1.  **语义化封装**：lark-cli将超过2500个原始API提炼成200多条AI友好的命令，有效避免了AI的“选择困难症”，降低了Token消耗和出错率。
2.  **Agent Skills**：它提供了19个开箱即用的AI Agent核心技能（Skills），将常用操作打包成逻辑单元。
3.  **三层架构**：提供了“快捷命令 → API命令 → 原始API”的三层结构，满足了从快速操作到深度控制的所有粒度需求。

#### 对话中提及的其他技术细节
*   **lark-cli 与 CC (Claude Code) 的集成**：群友所说的“Lark CLI接CC”，指通过`npx skills add`命令安装其Skill后，开发者可直接通过Claude Code的自然语言指令来调用所有飞书功能。
*   **插件系统与Token Usage测算**：指开发者在为自己的“lambda coding agent”增加模块化扩展能力，并精确计算每次调用产生的Token消耗，以实现成本控制。
*   **打包问题**：指Python项目在分发时的痛点，例如如何处理依赖关系和跨平台兼容性。
*   **lambda coding agent**：指向两种可能，一是指一个特定的开源项目，二是指利用“Lambda 架构”实时处理流数据，赋予AI Agent实时决策能力。
*   **FlashIdea**：可能是对话中提到的、尚未被搜索到结果的个人开源项目，旨在快速捕捉和实现创意灵感。

#### 总结
结合“Harness”的概念来看，这段对话确实展现了一个前沿的AI工程实践：开发者们正在用 **Lark CLI这类“底层指令”作为执行单元**，而整个 **Agent本身（在云端或本地运行）则扮演了“任务规划与调度”的Harness角色**。当Agent的规划能力足够强时，它便能通过调用这些丰富的CLI指令，自动完成曾经需要一个团队才能完成的、跨多个应用系统的复杂任务。
