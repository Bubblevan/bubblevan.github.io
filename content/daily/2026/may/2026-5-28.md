各位老师好，我是包博文，我的毕业设计题目是《面向双系统视觉语言导航模型的低时延推理加速设计与实现》。

我的汇报主要分为四部分：研究背景、技术方案、结果分析以及结论与展望。

首先介绍研究背景。导航一直是自主系统的一项基本能力，传统导航更关注定位、路径规划和避障。但是随着视觉语言模型进入具身智能任务，机器人不仅需要移动到目标位置，还需要根据语言指令和视觉观测进行高层语义理解。这里就出现了一个新的系统问题：高层视觉语言推理通常是间歇性的、低频的，并且推理时延较高；但底层控制必须在机器人移动和环境变化时持续运行。尤其在动态环境中，如果模型看到的场景和实际执行时的场景已经发生变化，那么高层语义决策即使本身合理，也可能因为延迟过大而无法落地。

本文的研究对象是 DualVLN 这类双系统视觉语言导航模型。DualVLN 采用类似“慢速高层规划”和“快速低层反应”的双系统结构。基于大模型的 System 2 接收导航指令、历史图像和当前图像，可以输出离散动作，也可以输出像素目标和潜在目标；基于Diffusion Transformer 的 System 1 则根据这些高层结果生成一小段局部连续轨迹，最后转化为底层动作。这类模型和普通大语言模型推理不同，它的输出不只是文本 token，还包括 System 1 后续要消费的中间表示。这也是本文后续做推理加速时必须额外处理的问题。

接下来介绍路线筛选与问题收敛的过程。前期我关注的是动态环境导航中的响应问题，测试过窄走廊、过门和人群等动态场景。以右侧示例为例，在正常步速下，机器人面对迎面行人时可能出现反应迟滞甚至冻结；只有在行人移动非常慢时，才偶尔表现出避让行为。这个现象说明，问题不只是模型有没有语义避让意图，而是这个意图能不能及时转化成有效动作。由于本地部署的模型输出主要是箭头、像素坐标和潜在表示，常规文本调试很难解释单次导航失败。因此我最终将问题收敛为一个更可度量的系统侧问题：降低高层推理时延、减少传输开销，并稳定完成双系统之间的中间结果传递。

基于这个收敛，本文设计了一个 training-free 的低时延推理加速方案，也就是不重新训练模型，而是从推理执行路径上进行优化。整体架构分为四层。第一层是任务请求层，负责接收导航指令、历史帧、当前帧和会话信息。第二层是输入接入与传输优化层，主要通过共享内存图像传输、多模态输入打包和输入构造来减少图像传输与序列化开销。第三层是状态化推理执行层，也是核心层，负责会话状态管理、请求调度与继续生成、规划结果适配以及执行后端适配。第四层是结果输出与下游消费层，将离散动作、像素目标和潜在目标统一交给 System 1 进行局部轨迹生成。这个设计的重点是把原本零散的模型调用，改造成一个面向导航闭环的状态化推理执行流程。

更具体地看状态化执行流程，系统接收观测后，会先判断最近一次高层结果是否仍然可复用。如果缓存有效，就不重新触发完整高层推理，而是优先复用已有结果；如果需要重新规划，则进入 System 2 高层推理。推理完成后，系统会根据输出类型进入不同解析路径：如果是离散动作，就直接向下游传递；如果是像素目标，则进一步提取潜在表征，并将结果封装为统一的 Planner Result，再交给 System 1 生成局部轨迹。这样做的目的，是在导航会话中管理结果有效性和中间表示，而不是每一步都把高层模型当作一次孤立调用。

下面是实验设置。本文主实验基于 Habitat VLNCE R2R 环境开展。这个基准以室内场景中的视觉语言导航为核心任务，整体上属于静态环境为主的标准 VLN benchmark。智能体需要根据自然语言指令在室内空间中到达目标位置，并在过程中持续调用高层规划和低层控制模块。评价指标分为两类：一类是导航指标，包括 SR、SPL、OSR 和 NE；另一类是系统时延指标，包括单个 episode 平均耗时、System 2 单步推理时延以及高层推理结果到低层执行之间的控制间隙。

在正式结果之前，我还做过一个预实验：Prefix Caching。初始设想是复用稳定文本前缀，降低 prefill 阶段的重复计算。实验发现，文本前缀在文本内部确实占比较高，但在完整多模态输入中只占约 3.86%。真正的大头是多图像 token，而不是文本前缀。因此，仅做文本 prefix caching 很难解决 VLN 模型的主要 prefill 压力。同时，单纯使用原生 vLLM 适配也无法直接重建 DualVLN 所需要的传输语义和中间结果接口。

接下来是主结果。从表中可以看到，相比 DualVLN 基线，本文的 VLN 推理加速方案将 System 2 平均时延从 507.98 毫秒降低到 305.87 毫秒；P99 长尾延迟从 672.00 毫秒降低到 399.90 毫秒；控制间隙从 322.47 毫秒降低到 208.63 毫秒；有效低层执行频率从 2.44 Hz 提升到 3.45 Hz。与此同时，SR、SPL 和 OSR 相比原始 DualVLN 只有小幅波动，明显改善了双系统闭环的时间结构。

为了确认导航指标的小幅波动不是由模型行为分布明显改变导致的，我进一步统计了 System 2 和 System 1 的行为一致性。三组配置下，像素目标产出率都约为 0.32，离散动作输出率约为 0.68，潜在表征生成成功率为 100%。同时，System 1 的触发率与 System 2 像素目标产出率基本匹配，单次触发平均产生约 11 步动作，说明低层轨迹展开规模也基本一致。因此可以认为，本文的推理加速主要改变的是系统时间结构，而没有明显改变高层输出类型分布和低层触发逻辑。

然后看传输开销分解。在通用 vLLM 适配路径中，客户端消息编码耗时达到 371.77 毫秒，HTTP 传输耗时也很高，而服务端解码只有 33.84 毫秒。这说明初始路径的额外开销主要集中在客户端请求体构造和多帧图像传输过程中。本文通过共享内存引用和轻量请求组织方式，减少了客户端和服务端之间的图像重复拷贝与编码开销。优化后，传输层不再是主要瓶颈，系统瓶颈重新回到模型推理本身。

对于在线闭环系统来说，只看均值是不够的，因为少量长尾请求也可能导致底层控制等待。因此我进一步分析了长尾稳定性。优化后，P50 从 508.9 毫秒降至 301.0 毫秒，P99 从 672.0 毫秒降至 399.9 毫秒。也就是说，本文方案不仅改善了平均情况，也压缩了长尾请求的时延范围，从而提高了闭环运行的稳定性。

此外，我也对 System 1 局部轨迹生成路径做了补充优化。这里主要有两类缓存：第一类是 Condition Cache，用于复用扩散过程中不变的 encoder-side 条件表征；第二类是 Cross-Attention KV Cache，用于复用 cross-attention 中由条件输入决定的 K/V。实验中两类 cache 合计节省约 18.01 毫秒，占 S1 总耗时约 13.5%。这说明 S1 侧缓存是有效的局部优化，但从整条闭环链路来看，其幅度有限，主优化方向仍然是 System 2 高层推理路径。

最后总结一下本文工作。第一，本文明确了双系统视觉语言导航中高层低频推理和低层高频执行之间的协同与时延问题。第二，本文提出并实现了一套 training-free 的低时延推理加速方案，不改变模型权重，而是从输入传输、状态化执行、中间结果封装和后端适配等方面优化执行路径。第三，在闭环评测中，System 2 平均时延从 507.98 毫秒降低到 305.87 毫秒，有效低层控制频率从 2.44 Hz 提升到 3.45 Hz。后续工作可以进一步探索更细粒度的多模态 token 压缩、更强的异步执行机制，以及更完整的真实动态环境评测。

我的汇报结束，请各位老师批评指正。
