# 10. x402
StablePay 现在要做的不是一个传统电商支付按钮，而是一个面向 AI Agent 的支付链路。

传统支付里，用户会看到页面、点击按钮、跳转收银台、扫码或者输入密码。
但 Agent 没有人类 UI，它更适合面对一种协议化流程：

```text
请求资源
  ↓
资源方说：需要付款
  ↓
Agent 读取付款要求
  ↓
Agent 调用钱包或支付网关付款
  ↓
付款后重试请求
  ↓
资源方验证付款并返回内容
```
x402 做的就是这件事：把“需要付款”这个信号变成 HTTP 世界里可以被机器理解的标准流程。

在 StablePay 里，x402 不只是一个文档概念。它会贯穿：
```text id="74trjy"
OpenClaw Plugin
  ↓
Merchant Server
  ↓
StablePay Gateway
  ↓
Payment Service
  ↓
Blockchain Adapter
  ↓
Solana / USDC
```
## 10.1 从一个最小愿望开始
我希望 Merchant Server 有一个付费商品接口：
```
GET /api/v1/products/report-001/execute
```
如果 Agent 没付款，我不想直接返回 403，也不想返回普通错误。

因为资源确实存在，Agent 也不是没有身份。真正的问题是 **你需要先付款，才能拿到这个资源**。

常见状态码里：
- 401 Unauthorized：你还没认证
- 403 Forbidden：你没有权限
- 404 Not Found：资源不存在
- 402 Payment Required：资源存在，但需要付款

StablePay 的商品接口适合 402，因为：
- 商品存在
- Agent 也可以识别自己
- 只是还没有支付这次资源访问费用

如果返回 403，Agent 只知道被拒绝，不知道下一步该怎么做；
如果返回 402，并附上 PaymentRequired，Agent 就知道下一步不是放弃，而是付款后重试。

这就是 x402 对 AI Agent 特别友好的地方。

所以最自然的 HTTP 状态码是 `402 Payment Required`！

最小响应可以长这样：
```http
HTTP/1.1 402 Payment Required
PAYMENT-REQUIRED: <base64-json>
Content-Type: application/json

{
  "code": "PAYMENT_REQUIRED",
  "message": "payment is required to access this resource"
}
```

这一步先不要考虑 Solana 怎么签名，也不要考虑 fee payer。
先把资源方的语义表达清楚：
> 这个资源不是不存在，也不是禁止访问，而是需要付款。

## 1.2 x402
x402 很容易被误解成一个支付网关。实际上它更像协议层，我们做的属于 Facilitator。

它规定的是：
- 资源方如何告诉客户端：这个资源需要付款
- 客户端如何携带付款授权重试请求
- 服务端如何返回支付或结算结果

它不强制你必须用某一家支付服务，也不限定必须只有一种链。
它只是把“请求资源 → 需要付款 → 付款 → 重试请求”的流程标准化。

在 StablePay 里：
- x402：定义 HTTP 支付协议
- StablePay Gateway：负责支付验证和支付提交
- blockchain-adapter：负责构建、提交、查询链上交易
- Solana：最终结算网络
- USDC：支付资产
### 1.2.1 三位角色
最小 x402 流程里可以抽象出三个角色：
- Client：客户端，StablePay 里是 OpenClaw Agent / Plugin
- Server：资源服务器，StablePay 里是 Merchant Server
- Facilitator：支付协调方，StablePay 里是 StablePay Gateway

Server 不一定要自己直接上链。它可以把支付验证、提交、结算交给 Facilitator。

这正是我们要做的分工：
- Merchant Server：商家卖什么，多少钱，付给谁
- StablePay Gateway：我来处理支付请求和验证购买状态
- Blockchain Adapter：我来做 Solana 交易构建、签名补贴、广播、状态查询

### 1.2.2 x402 v2
项目早期可能会看到旧式结构：
```json
{
  "accepts": [
    {
      "scheme": "exact",
      "network": "solana:...",
      "maxAmountRequired": "2000000",
      "payTo": "...",
      "asset": "...",
      "resource": "..."
    }
  ]
}
```
这种结构的问题是：
- resource 信息太弱，像是附属字段
- 金额字段 maxAmountRequired 命名偏旧

v2 更清楚地把 PaymentRequired 拆成：
```
x402Version
resource
accepts
extensions
```
这对 StablePay 很重要，因为我们不是只买一个 API 调用，而是要支持更多付费商品如报告、小礼品、技能、数据接口、Agent 服务等等

这些东西都可以抽象成“付费资源”。所以 `resource` 独立出来，对未来扩展更自然。

除此之外，v2 的 Header 更清楚：
- PAYMENT-REQUIRED：服务端告诉客户端需要什么付款
  - 服务端 → 客户端，你需要付什么
- PAYMENT-SIGNATURE：客户端带着付款授权或签名重试请求
  - 客户端 → 服务端，我已经授权/签名
- PAYMENT-RESPONSE：服务端返回支付处理结果
  - 服务端 → 客户端，支付处理结果

这比旧式 `X-*` Header 更容易理解方向，对我们自研插件很重要。

如果 Plugin、Gateway、Merchant Server 各自理解不同 Header，就会出现“我付款了但商家不认”的问题。

### 1.2.3 响应体
x402 v2 的 PaymentRequired 不应该只是一句“你要付钱”。

它应该包含一份机器可读的付款要求。

最小结构可以这样理解：
```json
{
  "x402Version": 2,
  "error": "PAYMENT-SIGNATURE header is required",
  "resource": {
    "url": "https://merchant.example.com/api/v1/products/report-001/execute",
    "description": "Access to premium report",
    "mimeType": "application/json"
  },
  "accepts": [
    {
      "scheme": "exact",
      "network": "solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1",
      "amount": "2000000",
      "asset": "USDC_MINT_ADDRESS",
      "payTo": "SELLER_WALLET_ADDRESS",
      "maxTimeoutSeconds": 300,
      "extra": {
        "currency": "USDC",
        "productId": "report-001",
        "skillDid": "did:solana:seller111"
      }
    }
  ]
}
```

这里先抓住几个核心字段：
```
x402Version：协议版本，v2 就写 2
resource：被保护的资源是什么
accepts：服务端接受哪些支付方式
scheme：支付方案，最小版本先用 exact
network：在哪条链或网络上支付
amount：最小单位金额
asset：支付资产，例如 USDC mint
payTo：收款方地址
extra：StablePay 自己需要的扩展信息
```
最重要的变化是：v2 里金额字段应该是 `amount`，而不是旧式 `maxAmountRequired`。
资源信息也应该放在独立的 `resource` 对象里，而不是只藏在 accepts 某个字段里。

---

## 1.3 为什么 `amount` 要用 minor units？

如果商品价格是：

```text id="48q9gh"
2.00 USDC
```

那么支付字段里不能写：

```json id="4zv82z"
"amount": "2.00"
```

而应该写：

```json id="e5l1x2"
"amount": "2000000"
```

因为 USDC 常见精度是 6 位小数：

```text id="4ntn9x"
1 USDC = 1,000,000 minor units
2 USDC = 2,000,000 minor units
0.000001 USDC = 1 minor unit
```

这也是为什么 Merchant Server 的 Domain 层要有 Money 值对象。

`"2.00"` 是展示金额。
`"2000000"` 是链上支付和 x402 合约里更稳定的金额表达。

这一步不要用 float64 随便乘：

```go id="nctwmx"
amountFloat, _ := strconv.ParseFloat("2.00", 64)
minor := int64(amountFloat * 1000000)
```

金额转换要用字符串或整数逻辑做。

## 1.4 最小 Go 结构体：先表达 x402 v2 合约

在 Merchant Server 的 Domain 层，可以先定义核心业务合约：

```go id="vsn85l"
type PaymentRequired struct {
    X402Version int                   `json:"x402Version"`
    Error       string                `json:"error,omitempty"`
    Resource    ResourceInfo          `json:"resource"`
    Accepts     []PaymentRequirements `json:"accepts"`
    Extensions  map[string]any        `json:"extensions,omitempty"`
}

type ResourceInfo struct {
    URL         string   `json:"url"`
    Description string   `json:"description,omitempty"`
    MimeType    string   `json:"mimeType,omitempty"`
    ServiceName string   `json:"serviceName,omitempty"`
    Tags        []string `json:"tags,omitempty"`
}

type PaymentRequirements struct {
    Scheme            string         `json:"scheme"`
    Network           string         `json:"network"`
    Amount            string         `json:"amount"`
    Asset             string         `json:"asset"`
    PayTo             string         `json:"payTo"`
    MaxTimeoutSeconds int            `json:"maxTimeoutSeconds,omitempty"`
    Extra             map[string]any `json:"extra,omitempty"`
}
```

这里要注意一件事：

> 这个结构不是普通 Response DTO，而是“付多少钱拿什么资源”的业务合约。

即使未来不用 HTTP，而是用 gRPC、MCP 或 A2A，这个合约仍然成立。
所以它更靠近 Domain 层。

---

## 1.5 Header 编码属于 Adapter 层

虽然 PaymentRequired 是业务合约，但 HTTP Header 是传输细节。

HTTP 里可能会这样传：

```http id="orwu1p"
PAYMENT-REQUIRED: eyJ4NDAyVmVyc2lvbiI6Mi...
```

这个 Header 的值本质是：

```text id="u4cmtg"
PaymentRequired JSON
  ↓
JSON Marshal
  ↓
Base64
  ↓
写入 PAYMENT-REQUIRED Header
```

这一步属于 Adapter 层，不属于 Domain 层。

Domain 层负责回答：

```text id="g0ka56"
这个资源要多少钱？
付给谁？
买什么？
用什么资产？
```

Adapter 层负责回答：

```text id="j70zjh"
这个业务合约如何通过 HTTP 402 返回？
Header 名叫什么？
Base64 怎么编码？
Body 怎么写？
```

所以代码边界应该是：

```text id="hz2a94"
Domain:
  BuildPaymentRequiredV2(product, config, resource)

Adapter:
  EncodePaymentRequiredHeader(required)
  c.Header("PAYMENT-REQUIRED", encoded)
  c.JSON(402, body)
```

---

## 1.6 最小 Handler 伪代码

商品执行接口可以先写成这样：

```go id="0bvu40"
func (h *ProductHandler) Execute(ctx context.Context, c *app.RequestContext) {
    productID := c.Param("id")
    agentDID := c.Query("agent_did")

    result, err := h.productAppService.ExecutePurchase(ctx, productID, agentDID)
    if err != nil {
        c.JSON(500, map[string]string{"message": err.Error()})
        return
    }

    if result.PaymentRequired != nil {
        encoded, err := EncodePaymentRequiredHeader(result.PaymentRequired)
        if err != nil {
            c.JSON(500, map[string]string{"message": "failed to encode payment requirement"})
            return
        }

        c.Header("PAYMENT-REQUIRED", encoded)
        c.JSON(402, map[string]any{
            "code": "PAYMENT_REQUIRED",
            "message": "payment is required to access this resource",
        })
        return
    }

    c.JSON(200, result.Content)
}
```

这段代码里有一个非常重要的分工：

```text id="w17jtl"
Handler 不判断商品价格；
Handler 不拼 Solana 交易；
Handler 不调用链；
Handler 只是把 AppService 返回的支付要求翻译成 HTTP 402。
```

---

## 1.7 Agent 侧的最小流程

OpenClaw Plugin 或 Agent 看到 402 后，应该这样做：

```text id="vm11gg"
1. 请求商品执行接口
2. 收到 402
3. 读取 PAYMENT-REQUIRED
4. Base64 解码成 PaymentRequired JSON
5. 选择 accepts[0]
6. 调用 stablepay_pay_via_gateway
7. 支付成功后拿到 tx_hash / proof / signature
8. 带 PAYMENT-SIGNATURE 重试原请求
9. Merchant Server 验证购买状态
10. 返回商品内容
```

这个流程是 x402 的核心价值。

Agent 不需要网页，不需要重定向，不需要扫码。
它只要理解 HTTP 402 和 PaymentRequired，就能完成自动支付。


# 二、八股概念基础知识点：x402 到底是什么？



---

## 2.6 x402 和 StablePay DID 的关系

x402 本身关心的是支付协议。
StablePay 还需要知道谁在买。

所以请求里通常需要：

```text id="4e9q4q"
agent_did
skill_did / product_id
wallet_address
```

`agent_did` 用来表示购买者。
`skill_did` 或 `product_id` 用来表示被购买资源。
`wallet_address` 用来做链上支付和验证。

这部分属于 StablePay 的扩展，不是 x402 本身的全部。

因此在 v2 里，可以把 StablePay 自己的字段放到：

```json id="tfuf2d"
"extra": {
  "agentDid": "did:solana:buyer111",
  "skillDid": "did:solana:seller111",
  "productId": "report-001",
  "currency": "USDC"
}
```

或者放到 `extensions` 里，具体取决于我们内部约定。

关键是：不要把 StablePay 扩展字段伪装成 x402 标准字段。

---

## 2.7 x402 的安全边界

x402 不是魔法。

它让支付协商标准化，但仍然要处理很多安全问题：

```text id="qa1r0x"
PaymentRequired 不能被篡改
amount 必须以商户服务端为准
payTo 必须来自可信配置
asset 必须来自可信配置
resource 必须和被访问接口绑定
付款授权不能被无限重放
tx_hash 不能只靠客户端自报
Merchant Server 必须向 Gateway 或链上验证
```

所以 Merchant Server 不应该看到客户端说：

```json id="y9gfpi"
{"paid": true}
```

就相信它。

正确逻辑应该是：

```text id="94rhlt"
客户端带 PAYMENT-SIGNATURE 或 payment proof 重试
  ↓
Merchant Server 调 StablePay Gateway 验证
  ↓
Gateway 根据 agent_did + product/skill + tx_hash 查询支付状态
  ↓
确认 purchased=true 后才返回内容
```

这也是为什么 `blockchain-adapter` 后面的交易状态查询很重要。
支付不是“提交了交易”就一定完成。链上还要确认，甚至可能失败。

# 三、StablePay 落地：x402 只管“要付款”，blockchain-adapter 负责“真转账”

## 3.1 StablePay 的完整链路

把整个链路串起来：

```text id="hyz9i9"
Agent 请求 Merchant Server 的付费资源
  ↓
Merchant Server 返回 402 + PAYMENT-REQUIRED
  ↓
Agent 解析 x402 v2 PaymentRequired
  ↓
Agent 调 stablepay_pay_via_gateway
  ↓
Gateway 进入 Payment Service
  ↓
Payment Service 调 Blockchain Adapter
  ↓
Blockchain Adapter 构建 / 提交 Solana USDC 交易
  ↓
Gateway 记录 tx_hash 和购买关系
  ↓
Agent 重试 Merchant Server
  ↓
Merchant Server 调 Gateway verify
  ↓
返回商品内容
```

这说明 x402 在 StablePay 里主要连接两端：

```text id="ee3t3x"
上游：Agent 如何知道该付款
下游：Gateway 如何知道该执行哪笔链上支付
```

但真正的链上复杂度在 blockchain-adapter。

---

## 3.2 为什么 blockchain-adapter 经历了很多风雨？

因为“让 Agent 支付 USDC”听起来简单，实际上在 Solana 上会遇到一堆细节。

最开始可能会以为：

```text id="3g3wy2"
我只要知道 from、to、amount，然后服务端发一笔转账就行。
```

但很快发现不对。

如果服务端直接构建并签名交易，代币会从谁的账户扣？
如果买家的钱包在 Agent 本地，服务端没有买家的私钥，怎么替买家签？
如果让买家本地签名，那交易里的 fee payer 是谁？
如果平台想补贴 gas，fee payer 又该怎么加入？
如果收款方没有 ATA，谁来创建？
如果 blockhash 过期，交易会不会失败？
如果 `getTransaction` 查不到，是失败还是 RPC 索引延迟？
如果客户端传了恶意 signed tx，热钱包能不能被诱导签错交易？

这些问题不是 x402 字段能解决的。
它们属于 Solana 交易执行层。

---

## 3.3 从“一步转账”到“两步交易”的演进

最朴素的接口可能是：

```text id="n1yg3y"
TransferStableCoin(to, amount)
```

服务端收到请求后直接转账。

这种模式只适合一种情况：

```text id="0ukelx"
钱本来就在服务端热钱包里
服务端有权花这笔钱
热钱包既是 token owner，也是 fee payer
```

但 Agent 支付不是这个模型。
钱在 Agent 自己的钱包里，服务端不应该也不能拥有 Agent 私钥。

所以后面会演进成两步：

```text id="cq17cu"
1. BuildUnsignedTransaction
   服务端构建交易，指定 fee payer、to、amount、asset、blockhash

2. SubmitSignedTransaction
   Agent 本地签名后提交，服务端热钱包补签 fee payer，再广播
```

这就是你们代码里为什么会出现：

```text id="rwmut6"
BuildUnsignedTransaction
SubmitSignedTransaction
SignedTxBase64
UnsignedTxBase64
hotWallet.SignBase64Transaction
ValidatePartiallySignedTransaction
```

这些不是凭空复杂化，而是被“买家自持钱包 + 平台补贴 gas + Solana 多签交易”逼出来的。


## 3.4 fee payer 是整个链路的关键坑

Solana 交易里，fee payer 通常是 message account keys 的第一个账户。

如果 StablePay 平台想补贴 gas，就要让热钱包作为 fee payer。

但买家的 USDC 还是要从买家自己的 ATA 扣。
所以一笔交易里至少有两个重要签名角色：

```text id="kelpc0"
买家：SPL Token owner / authority，授权扣自己的 USDC
热钱包：fee payer，支付 SOL 手续费
```

这就是 partial signing 的来源。

正确方向是：

```text id="ms9zwc"
服务端构建交易：fee payer = hot wallet
  ↓
Agent 本地钱包签名：签 token owner
  ↓
服务端热钱包补签：签 fee payer
  ↓
广播交易
```

如果服务端没有校验 fee payer，就可能被恶意交易诱导热钱包乱签。
所以 hot wallet 签名前必须确认：

```text id="d88p59"
这笔交易的 fee payer 真的是热钱包自己
```

你们代码里 `SignBase64Transaction` 会反序列化交易，并检查第一个 account key 是否等于热钱包公钥。这个校验非常关键。

---

## 3.5 为什么不能在没有 signed_tx_base64 时随便服务端构建？

你们 `TransferCmdService` 里已经做了一个重要防线：

```text id="39h0b6"
如果没有 signed_tx_base64，
只有 fromAddress 为空或 fromAddress 等于 hot wallet 时，
才允许走服务端自建交易。
```

这背后的坑是：

```text id="qh6tte"
如果 from_wallet 是买家地址，
但服务端没有买家私钥，
服务端自建交易是不可能合法花买家 USDC 的。
```

更危险的是，如果实现写错，可能会变成：

```text id="f0f8dq"
请求里写的是买家 from
实际扣的是热钱包 ATA
```

所以你们后来加了拒绝逻辑：

```text id="kr6pa9"
signed_tx_base64 is required when from_wallet is not the custodial hot wallet
```

这就是一次很典型的后端支付风控修正。

它说明 blockchain-adapter 不能只看接口字段，还必须明确“谁有权花谁的钱”。

---

## 3.6 ATA 是 Solana 支付绕不开的细节

在 Solana 上，USDC 不是直接存在钱包地址上，而是存在某个 mint 对应的 Associated Token Account，也就是 ATA。

所以从 A 给 B 转 USDC，实际涉及：

```text id="v9nt1x"
A 的 USDC ATA
B 的 USDC ATA
USDC mint
Token Program
```

如果 B 还没有 USDC ATA，转账会失败。

所以 blockchain-adapter 构建 SPL Token 转账时要考虑：

```text id="zxb0xw"
计算 fromATA
计算 toATA
检查 toATA 是否存在
不存在则加 create ATA 指令
再加 token transfer 指令
```

这解释了为什么链上执行层比 x402 字段复杂很多。

x402 只会说：

```text id="osyu5t"
payTo = seller wallet
asset = USDC mint
amount = 2000000
```

但 blockchain-adapter 要把这句话翻译成 Solana 真正能执行的指令。

---

## 3.7 blockhash 是另一个常见大坑

Solana 交易需要 recent blockhash。

如果客户端签的是一个旧 blockhash，提交时可能报：

```text id="5ypijr"
blockhash not found
```

或者 preflight 失败。

所以你们的 SolanaGateway 在发送交易前会检查：

```text id="imgwok"
IsBlockhashValid
```

如果发现 blockhash 不在当前 RPC 或已经过期，就直接返回更清晰的错误。

这也是为什么 BuildUnsignedTransaction 和 SubmitSignedTransaction 最好在同一个网络、同一个 RPC 上配合。

否则可能出现：

```text id="prn55l"
客户端拿 devnet A 的 blockhash 签名
服务端拿 devnet B 或 mainnet 的 RPC 提交
最终报 blockhash invalid
```

这类问题看起来像“链不稳定”，其实经常是网络、RPC、blockhash 生命周期不一致。

---

## 3.8 为什么交易提交后还要查状态？

`SendTransaction` 成功返回 tx hash，不等于购买已经完成。

它只说明交易被 RPC 接收或广播成功。

后面还可能：

```text id="0hd5i3"
pending
confirmed
finalized
failed
```

所以 blockchain-adapter 需要：

```text id="d9q97d"
GetTxStatus
WaitForConfirmation
GetTransaction
GetSignatureStatuses fallback
```

你们代码里有一个很现实的处理：
当 `getTransaction` 还查不到时，会 fallback 到 `getSignatureStatuses`。

这是因为 RPC 索引可能滞后。
交易已经 confirmed，但 `getTransaction` 暂时查不到，并不一定代表失败。

对于 x402 来说，这个区别很重要：

```text id="iyf8pm"
pending：Agent 可能需要等待或重试
confirmed：Merchant 可以放行
failed：Payment 必须失败
```

---

## 3.9 错误码映射为什么很重要？

在支付系统里，错误不能全都变成 500。

不同错误对上层含义完全不同：

```text id="7ncw05"
INVALID_PARAMETERS：客户端请求或交易格式错
INSUFFICIENT_BALANCE：买家余额不足
BLOCKCHAIN_NETWORK_ERROR：链上 RPC、preflight、blockhash、simulation 问题
RESOURCE_NOT_FOUND：查询资源不存在
INTERNAL_SERVER_ERROR：真正内部错误
```

如果错误码映射太粗，会影响 Gateway 和 Agent 的决策。

例如：

```text id="67zu94"
余额不足：Agent 应该提示充值或换钱包
blockhash 过期：Agent 应该重建交易
参数错误：插件代码可能有 bug
RPC 网络错误：可以稍后重试
```

你们 `TransferRPCAdapter` 里对错误字符串做了更细的映射，这就是“经历风雨”后的工程化修补。

## 面试讲法
x402 是一个把 HTTP 402 Payment Required 真正用起来的支付协议。
传统支付依赖网页跳转和人工交互，但 AI Agent 没有人类 UI。
x402 的做法是：Agent 请求资源，服务端返回 402 和机器可读的 PaymentRequired；
Agent 解析后调用钱包或支付网关付款，再带着付款授权重试请求。
在 StablePay 里，Merchant Server 负责返回 x402 v2 PaymentRequired，
StablePay Gateway 负责支付验证和调度，blockchain-adapter 负责真正构建、
签名、提交 Solana USDC 交易。


### 7.1 blockchain-adapter 重构
一开始我们以为支付就是 from、to、amount 三个字段。
但真正接 Solana 后发现，Agent 自持钱包和平台补贴 gas 会带来多签问题：
买家要作为 token owner 签名，热钱包要作为 fee payer 补签。
所以我们从单步 Transfer 演进成 BuildUnsignedTransaction + SubmitSignedTransaction。
服务端先构建包含 fee payer 的未签名交易，客户端本地签名后提交，
热钱包校验自己确实是 fee payer，再补签并广播。
同时我们还处理了 ATA 创建、blockhash 过期、RPC 确认滞后和错误码映射等问题。
这部分复杂度不属于 x402 协议本身，而属于链上执行层。

### 7.2 分层边界
我把 x402 和 blockchain-adapter 分成两层看。

第一层是协议层：Merchant Server 返回 x402 v2 PaymentRequired，告诉 Agent
要访问哪个 resource，需要付多少 amount，使用哪个 asset，付给哪个 payTo。
这部分是“付多少钱拿什么资源”的业务合约。

第二层是执行层：StablePay Gateway 把这个支付要求变成具体链上动作，
blockchain-adapter 负责 Solana 交易，包括构建交易、客户端部分签名、
热钱包补签 fee payer、发送交易和查询状态。

这样分层后，x402 v1/v2 的兼容主要发生在 Merchant/Gateway，
blockchain-adapter 只处理标准化后的链上参数，不需要理解商品和 x402 字段。
