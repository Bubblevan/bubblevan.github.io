# 13. 从 verification-service 到 RocketMQ
从 `verification-service` 讲起。它的业务逻辑是，给我一个 `agent_did` 和 `skill_did`，我告诉你这个 Agent 有没有买过这个 Skill。

但这个小问题往后长，就会自然长出几个后端开发里绕不开的话题：
- 为什么链上交易成功不等于业务上已经购买成功；
- 为什么需要消息队列；
- RocketMQ 的 NameServer、Broker、Topic、Consumer Group 到底在项目里起什么作用；
- 为什么一次 Pod 重启、一次 Rollout，可能让“已经付款”继续返回 402。

## 13.1 初始判断
最早我想解决的问题很朴素：
```text
Agent 访问受保护资源
  ↓
商家问 StablePay：这个 Agent 买过这个 Skill 吗？
  ↓
买过：返回 200
没买过：返回 402 Payment Required
```
所以 `verification-service` 的第一版并不需要复杂。

它只需要有一张购买记录表：
```go
type PurchaseRecord struct {
    gorm.Model
    AgentDid string `gorm:"type:varchar(128);uniqueIndex:uk_agent_skill"`
    SkillDid string `gorm:"type:varchar(128);uniqueIndex:uk_agent_skill"`
    TxId     string `gorm:"type:varchar(64);index"`
}
```
这张表背后的业务语义很重要。

它不是链上流水表，也不是 payment-service 的交易表。  
它表达的是：
> 对某个 Agent 来说，某个 Skill 已经被业务系统承认为“已购买”。

所以查询接口也很直接：
```go
func (s *VerificationServiceImpl) VerifyPurchase(
    ctx context.Context,
    req *verification_service.VerifyPurchaseRequest,
) (*verification_service.VerifyPurchaseResponse, error) {
    resp := verification_service.NewVerifyPurchaseResponse()
    var record PurchaseRecord

    result := DB.
        Where("agent_did = ? AND skill_did = ?", req.AgentDid, req.SkillDid).
        First(&record)

    if result.Error == nil {
        resp.Purchased = true
        txId := common.TxId(record.TxId)
        resp.TxId = &txId
        resp.Base = &common.BaseResp{Code: 0, Message: "success"}
    } else {
        resp.Purchased = false
        resp.Base = &common.BaseResp{Code: 10001, Message: "record not found"}
    }

    return resp, nil
}
```
链上交易成功只能说明资产转移发生了。  

而 `VerifyPurchase` 返回 true，说明支付事件已经穿过了 payment-service、MQ、verification-service，最终落成了业务记录。

### 13.1.1 为什么不能让商家服务自己记账
我一开始也可以把购买记录直接放在商家服务里。

比如 merchant-backend 收到支付回调后，自己写一张 `purchased_products` 表。

但这样很快会遇到几个问题：
- 一个 Agent 可能购买多个 Skill，不一定只对应某个商家页面；
- payment-service 是支付状态的源头，merchant 只是资源提供者；
- 后续如果有多个商家、多个 Agent 工具，都应该复用同一个购买验证能力；
- 链上支付确认是异步的，不能强迫商家请求线程一直等到最终态。

所以我把“是否购买过”收敛到 `verification-service`。

merchant-backend 只做一件事：
```text
请求 verification-service
  ↓
purchased=true：解锁资源
purchased=false：返回 402
```
这样一来，商家服务就不需要理解 payment-service 的内部状态机，也不需要直接监听链上交易。

这就是 StablePay 里 verification-service 的边界。
## 13.1.2 从同步查询到异步消费
只有查询是不够的。

因为 `purchase_records` 总要有人写进去。

payment-service 里有支付状态机。它轮询链上交易，当交易确认后，会把 Payment 从 `PENDING` 推进到 `CONFIRMED` 或 `COMPLETED`。

在状态变化后，它会发布支付事件：
```go
func (s *PaymentApplicationService) publishEvent(ctx context.Context, payment *entity.Payment) {
    event := &dto.MQPaymentEvent{
        TxID:        payment.TxID,
        AgentDID:    payment.AgentDID,
        SkillDID:    payment.SkillDID,
        AmountMinor: payment.AmountMinor,
        Currency:    vo.CommonCurrencyToString(payment.Currency),
        TxHash:      payment.TxHash,
        Status:      constants.PaymentStatusToString(payment.Status),
        Timestamp:   time.Now().Unix(),
    }

    if err := s.eventPublisher.PublishPaymentEvent(ctx, event); err != nil {
        s.logger.Error("failed to publish payment event", zap.Error(err))
    }
}
```
做一个思维实验。假如说是payment-service 在链上确认后，直接调用 verification-service 的 gRPC 接口，把记录写进去的话：
```
payment-service
  轮询链上交易 → 状态变成 COMPLETED
  ↓ 同步 RPC 调用
verification-service
  写 purchase_records
  ↓ 返回
merchant-backend 才能查到 true
```
很快暴露出几个问题：

1. **强耦合**：payment-service 必须知道 verification-service 的地址，并且依赖它的可用性。如果 verification-service 重启或升级，支付流程可能卡住。
2. **阻塞**：支付确认是异步事件，但写库操作却要等 verification-service 返回。万一 verification-service 慢，payment-service 的轮询线程也会被拖慢。
3. **重试困难**：如果写库失败（比如数据库连接断开），payment-service 需要自己实现重试逻辑，而且可能重复写入。

能不能让 payment-service **只负责把“支付成功”这件事说出来**，至于谁听、听了做什么，它不关心？

这就是消息队列（MQ）的原始动机——**解耦生产者和消费者，让事件驱动代替同步调用**。

payment-service 没有 **直接调用** verification-service 写库。它只是发布某个 tx_id 的支付状态变化了的一个事实。

verification-service 自己监听这个事实，决定要不要写 `purchase_records`。
```go
err = c.Subscribe("payment_events", consumer.MessageSelector{}, func(
    ctx context.Context,
    msgs ...*primitive.MessageExt,
) (consumer.ConsumeResult, error) {
    for _, msg := range msgs {
        tag := msg.GetTags()
        if tag != "" && tag != "payment_succeeded" {
            continue
        }

        var event PaymentSuccessEvent
        if err := json.Unmarshal(msg.Body, &event); err != nil {
            continue
        }

        record := PurchaseRecord{
            AgentDid: event.AgentDid,
            SkillDid: event.SkillDid,
            TxId:     event.TxId,
        }

        if err := DB.Create(&record).Error; err != nil {
            return consumer.ConsumeRetryLater, err
        }
    }

    return consumer.ConsumeSuccess, nil
})
```
这就是异步事件驱动的价值：
- payment-service 负责支付状态；
- verification-service 负责购买证明；
- 两者之间不需要同步 RPC 强耦合；
- 消费失败可以重试；
- 业务最终一致，而不是强一致。

> 我们选了 RocketMQ，因为它在阿里云环境里集成方便，而且支持事务消息（虽然当前版本还没用上）。
## 13.2 RocketMQ
链路是：
```text
payment-service
  轮询链上交易
  状态变成 COMPLETED
  发布 payment_events

RocketMQ
  存储并投递支付事件

verification-service
  消费 payment_events
  写入 purchase_records

merchant-backend
  每次执行资源前查询 VerifyPurchase
```
这里可以自然带出几个 RocketMQ 八股点。
### 13.2.1 NameServer：路由注册中心
用了 MQ 之后，payment-service 不再直接调用 verification-service，而是向 RocketMQ 发一条消息。
```go
// payment-service 发送支付成功事件
msg := primitive.NewMessage("payment_events", body).
    WithTag("payment_succeeded")
p.producer.SendSync(ctx, msg)
```
verification-service 启动一个消费者，订阅 `payment_events`，收到消息后写库。

但是，**payment-service 怎么知道 RocketMQ 的服务器在哪里**？  
verification-service 又怎么知道去哪里拉消息？

在 RocketMQ 里，这个“寻址”的角色叫 **NameServer**。

NameServer 是一个轻量级的路由中心。生产者和消费者启动时，只配置 NameServer 的地址（比如 `stablepay-rocketmq-nameserver:9876`），然后问 NameServer：

> “payment_events 这个 topic 的消息存在哪个 Broker 上？Broker 的 IP 是什么？”

NameServer 不会存储消息，它只维护一份**路由表**。

所以在我们的代码里，配置是这样的：
```yaml
rocketmq:
  name_servers:
    - stablepay-rocketmq-nameserver:9876
  topics:
    payment_events: payment_events
```
启动时，RocketMQ 客户端会从 NameServer 拉取路由信息，之后才能正常收发消息。

verification-service 里通过环境变量拿 NameServer：
```go
raw := strings.TrimSpace(os.Getenv("ROCKETMQ_NAMESERVER"))
if raw == "" {
    return []string{"127.0.0.1:9876"}
}
```
其职责不是存消息，而是告诉客户端：
- 某个 topic 在哪些 broker 上
- broker-a 当前地址是什么
- 这个 topic 有哪些 queue
> **NameServer 挂了怎么办？**  
> 答：客户端会缓存上一次的路由，短时间内还能工作，但新增 topic 或 Broker 变更会受影响。生产环境一般部署多个 NameServer 实例做高可用。

### 13.2.2 Broker：真正存消息的地方
NameServer 只是指路人，真正存储消息、投递消息的是 **Broker**。

在我们的 K8s 环境里，Broker 是一个独立的 Pod：
```yaml
containers:
  - name: broker
    image: stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/rocketmq:5.3.2
    command:
      - sh
      - -c
      - sleep 8 && sh /home/rocketmq/rocketmq-5.3.2/bin/mqbroker -c /home/rocketmq/rocketmq-5.3.2/conf/broker.conf
```
Broker 的配置文件里会 **指定 NameServer 地址** 并打开了自动建 topic：
```properties
brokerClusterName = DefaultCluster
brokerName = broker-a
brokerId = 0
namesrvAddr = stablepay-rocketmq-nameserver:9876
autoCreateTopicEnable = true
```

Broker 启动后，会自动向 NameServer 注册，告诉 NameServer：“我是 broker-a，我能处理哪些 topic。”

当 payment-service 发送消息时，客户端从 NameServer 拿到 Broker 地址，然后直接与 Broker 通信，把消息存入磁盘。verification-service 消费时，也是从 Broker 拉取消息。

**Broker 的职责**：
- 接收生产者发来的消息，持久化到 CommitLog。
- 维护 Consumer Queue，记录消息的偏移量。
- 处理消费者的拉取请求，投递消息。
- 定期向 NameServer 发送心跳，更新路由信息。

> 埋个伏笔：`autoCreateTopicEnable=true` 不是生产级保障。
> 它不能替代显式 topic 初始化。
> 尤其是在 K8S 里，如果 broker Pod 重建，而 broker 又没有挂持久化卷，那么很多运行期状态都可能丢。即使 broker 还活着，NameServer 里也可能没有业务 topic 的路由。

### 13.2.3 Topic：业务事件流的名字
`payment_events` 就是我们这条支付事件流的 topic。
```go
msg := primitive.NewMessage("payment_events", body)
```
Topic 是 RocketMQ 里最顶层的业务分类。所有支付相关的消息都发往这个 Topic，而退货消息可以发往另一个 Topic `refund_events`。

在我们的环境里，Broker 配置了 `autoCreateTopicEnable=true`，所以第一次发送消息时，Broker 会自动创建 Topic。
> 但生产环境建议显式创建，避免权限问题和自动创建的默认队列数不符合预期。

payment-service 创建 Producer 时把 topic 注入进去：
```go
mqProducer, err := mq.NewPaymentEventProducer(
    cfg.RocketMQ.NameServers,
    cfg.RocketMQ.ProducerGroup,
    cfg.RocketMQ.Topics["payment_events"],
    logger,
)
```
真正发送时：
```go
msg := primitive.NewMessage(p.topic, data).
    WithTag(tag).
    WithKeys([]string{event.TxID})

res, err := p.producer.SendSync(ctx, msg)
```
在 verification-service 里，消费者订阅时也是写同一个 Topic：

```go
c.Subscribe("payment_events", ...)
```
通过 Topic，我们把“支付事件”这个业务流从物理上隔离开，不同 Topic 的消息不会串。

这里有两个面试点：
- `Topic` 表示一类消息，比如支付事件；
- `Key` 可以用业务主键，比如 `tx_id`，方便排查消息；
- `Tag` 可以在同一个 topic 下进一步区分事件类型。

### 13.2.4 Tag：同一 topic 下的轻量分类
同一个 Topic 下，可能有多种消息类型：支付成功、支付失败、支付超时等。如果为每一种类型单独建一个 Topic，管理成本太高。

RocketMQ 提供了 **Tag** 机制。Tag 是依附于消息的一个标签，生产者在发送时可以指定：

```go
msg.WithTag("payment_succeeded")
```
消费者可以选择只订阅某个 Tag：
```go
// 只订阅支付成功的消息
consumer.Subscribe("payment_events", consumer.MessageSelector{
    Type:       consumer.TAG,
    Expression: "payment_succeeded",
})
```
比如说 payment-service 会把 `COMPLETED` 或 `CONFIRMED` 转成支付成功 tag：
```go
func eventTagFromStatus(status string) string {
    switch strings.ToUpper(strings.TrimSpace(status)) {
    case "CONFIRMED", "COMPLETED":
        return dto.ToMQEventTag(3)
    case "FAILED", "CANCELLED":
        return dto.ToMQEventTag(4)
    default:
        return ""
    }
}
```
verification-service 消费时只关心支付成功：
```go
tag := msg.GetTags()
if tag != "" && tag != "payment_succeeded" {
    continue
}
```
这样做的好处：
- 减少消费者收到的无关消息，提高效率。
- 同一个 Topic 可以承载多种事件类型，便于统一管理。
- 如果需要新增“支付超时”事件，只需要增加一个新的 Tag，不需要改动消费组的订阅逻辑（只要忽略它即可）。

我们的设计是：
```text
Topic：payment_events
Tag：payment_succeeded / payment_failed
Key：tx_id
Body：agent_did、skill_did、tx_hash、amount、status
```

### 13.2.5 Consumer Group：消费进度的归属
verification-service 是一个典型的事件消费者。如果我把这个服务扩容到 3 个 Pod，它们应该共同消费 `payment_events` 里的消息，每条消息只被其中一个 Pod 处理。

RocketMQ 用 **Consumer Group** 来实现这个能力。在同一个 Group 下的多个消费者实例，会自动负载均衡队列，协同消费。

verification-service 使用固定 group：
```go
consumer.WithGroupName("verification_group")
```
这意味着：
- 同一个 group 里的多个 consumer 共同消费；
- RocketMQ 会记录这个 group 的消费 offset；
- 如果消费失败，可以进入重试逻辑；
- `%RETRY%verification_group` 是这个 group 的重试 topic。

Group 的名称很重要，因为 RocketMQ 会以 `ConsumerGroup + Topic` 为维度，记录消费进度（offset）。

例如，`verification_group` 已经消费到了 `payment_events` 的第 100 条消息。如果 verification-service 重启，它会从第 101 条继续消费，不会重复处理已消费的消息。

如果同一个 Group 下有多个消费者实例，它们会分摊队列，共同推进 offset。如果某个实例挂了，RocketMQ 会重新分配队列给其他实例，保证消费不中断。
> **Consumer Group 和 Topic 的关系？**
> 答：一个 Group 可以订阅多个 Topic，但每个 Topic 的 offset 是独立存储的。一个 Topic 可以被多个不同 Group 订阅，每个 Group 有自己的 offset，互不干扰。

### 13.2.6 小结
| 代码 | 对应的 RocketMQ 概念 | 为什么需要它 |
|--------------|---------------------|-------------|
| `primitive.NewMessage("payment_events", body)` | Topic | 把支付事件和其他业务消息（如退款）隔离 |
| `msg.WithTag("payment_succeeded")` | Tag | 在同一个 Topic 下区分成功/失败事件，消费者按需过滤 |
| `producer.SendSync(ctx, msg)` | Producer | 发送消息的客户端角色 |
| `consumer.Subscribe("payment_events", ...)` | Consumer | 拉取并处理消息的客户端角色 |
| `consumer.WithGroupName("verification_group")` | Consumer Group | 管理消费进度，支持水平扩展和故障转移 |
| `rocketmq.name_servers: xxx:9876` | NameServer | 服务发现，让客户端找到 Broker |
| Broker Pod 和 `broker.conf` | Broker | 真正存储和投递消息的服务器 |

## 13.3 为什么“链上成功”后还是 402
开发的时候，这个问题遇到至少3次。

链上交易已经成功，USDC 已从 Agent 钱包转出，商家账户余额也增加了。但是再次请求商品时，仍然收到 402 Payment Required，/verify 接口返回 purchased=false。

直觉上，链上成功 ≈ 钱已到账 ≈ 应该能访问。但实际上，链上成功只是支付流程的终点，却不是购买证明写入的终点。中间还有一条异步链路：
```
链上交易确认
  ↓
payment-service 轮询到 COMPLETED
  ↓
发送 RocketMQ 消息 (topic: payment_events, tag: payment_succeeded)
  ↓
verification-service 消费消息
  ↓
写入 purchase_records 表
  ↓
merchant-backend 调用 VerifyPurchase 查询
  ↓
返回 true (200) 或 false (402)
```
只要中间任何一步断了，merchant 就会继续返回 402。
### 13.3.1 决策链路
面对“链上成功但没买过”的现象，我的排查策略是**从下游往上游推**：
1. 先看 verification-service 的日志——它是否收到了消息？
2. 如果没收到，再检查 RocketMQ —— topic 是否存在？Broker 是否正常？
3. 如果 MQ 正常，再看 payment-service 是否真的发布了消息。
别先怀疑商家服务。因为 merchant 只做查询。如果 verification 返回 false，它返回 402 是正常行为。

所以我先看 verification 日志：
```bash
kubectl -n zheda-agent logs --tail=100 deploy/stablepay-verification-service
```
- 第一类是 **topic 不存在**：
```text
query topic route from server error underlayError="topic not exist"
queryTopicRouteInfoFromServer return nil topic=payment_events
```
看到 `topic not exist` 后，我去查 RocketMQ 当前路由：

```bash
kubectl -n zheda-agent exec deploy/stablepay-rocketmq-broker -- \
  sh -c 'sh /home/rocketmq/rocketmq-5.3.2/bin/mqadmin topicRoute \
  -n stablepay-rocketmq-nameserver:9876 -t payment_events'
```
结果是：
```text
No topic route info in name server for the topic: payment_events
```
再查 topic 列表：
```bash
kubectl -n zheda-agent exec deploy/stablepay-rocketmq-broker -- \
  sh -c 'sh /home/rocketmq/rocketmq-5.3.2/bin/mqadmin topicList \
  -n stablepay-rocketmq-nameserver:9876'
```
只看到系统 topic，没有 `payment_events`。

- 第二类是 **consumer 还在访问旧 broker 地址**：
```bash
kubectl -n zheda-agent exec deploy/stablepay-rocketmq-broker -- \
  sh -c 'sh /home/rocketmq/rocketmq-5.3.2/bin/mqadmin clusterList \
  -n stablepay-rocketmq-nameserver:9876'
```
verification 还在打：
```text
pull message from broker error broker="10.112.124.116:10911"
underlayError="dial tcp 10.112.124.116:10911: i/o timeout"
```
而我查当前集群里 Nameserver 认识的 Broker 的真实 IP：
```text
当前 broker Pod IP: 10.55.180.174
verification 日志里的 broker: 10.112.124.116
```

这说明两个问题叠在一起：

- `payment_events` 在当前 NameServer 里没有路由；
- verification 客户端里还缓存着旧 broker 路由。

### 13.3.2 为什么 topic 会丢
我现在的判断是：这不是每次普通 Rollout 都必然发生，而是和 RocketMQ 的部署方式有关。

我们当时把 RocketMQ Broker 部署成了普通的 **Deployment**，并且**没有挂载持久化卷（PVC）**：

```yaml
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: broker
          image: rocketmq:5.3.2
          # 没有 volumeMounts 指向持久化存储
```

Broker 的动态数据（包括自动创建的 topic、消费进度等）都写在容器的临时文件系统里。一旦 Pod 因为滚动更新、节点迁移或手动重启而被重建，所有动态 topic 都会消失。即使 broker 重新启动后向 NameServer 注册了自己，`payment_events` 这个 topic 也不会自动恢复，因为它原来就是“运行时自动创建”的，不是显式初始化的基础设施。
```text
broker Pod 重建
  ↓
容器文件系统重置
  ↓
动态创建的业务 topic 可能丢失
  ↓
NameServer 里没有 payment_events route
  ↓
producer/consumer 查不到 topic
```
再加上我们之前因为错误 apply 了 `infra.yaml` 导致 RocketMQ 回滚过，broker Pod 被重建了。重建后，NameServer 里就没有了 `payment_events` 的路由信息。
### 13.3.3 为什么 verification 还在访问旧 Broker IP？
RocketMQ 客户端（生产者和消费者）在启动时会从 NameServer 拉取 topic 的路由表，并缓存在内存中。只要客户端进程不重启，它就会一直使用第一次拉到的 Broker 地址列表。

Broker Pod 被重建后，IP 变了。NameServer 里的路由表已经更新为新 IP（`10.55.180.174`）。但是 verification-service 的 Pod 并没有重启，它的客户端内存里还保留着旧 IP `10.112.124.116`。当它尝试从旧 IP 拉消息时，自然超时。

这就是为什么日志里会出现 `dial tcp 10.112.124.116:10911: i/o timeout`。

### 13.3.3. 临时修复方案
临时修复分两步。

第一步，重新创建 topic：
```bash
kubectl -n zheda-agent exec deploy/stablepay-rocketmq-broker -- \
  sh -c 'sh /home/rocketmq/rocketmq-5.3.2/bin/mqadmin updateTopic \
  -n stablepay-rocketmq-nameserver:9876 \
  -c DefaultCluster \
  -t payment_events \
  -r 4 -w 4'
```

第二步，重启 payment 和 verification，清理客户端缓存的旧 broker route：
```bash
kubectl -n zheda-agent rollout restart deployment/stablepay-payment-service
kubectl -n zheda-agent rollout restart deployment/stablepay-verification-service

kubectl -n zheda-agent rollout status deployment/stablepay-payment-service --timeout=120s
kubectl -n zheda-agent rollout status deployment/stablepay-verification-service --timeout=120s
```

然后观察日志：
```bash
kubectl -n zheda-agent logs --since=2m deploy/stablepay-verification-service
```

期望不再出现：

```text
topic not exist
dial tcp 10.112.124.116:10911: i/o timeout
```

之前那笔链上已经成功的交易，大概率不会自动补进 verification，因为当时 topic 不存在，success 事件没有成功进入 MQ。因此需要我们手动补齐历史数据：
1. 自动获取 MySQL Pod 名称
```bash
MYSQL_POD=$(kubectl -n zheda-agent get pod -l app=stablepay-mysql \
  --field-selector=status.phase=Running \
  -o jsonpath='{.items[0].metadata.name}')
```
2. 将 **链上tx_hash** 替换为实际链上交易哈希值后执行：
```bash
kubectl -n zheda-agent exec -i "$MYSQL_POD" -- mysql \
  --default-character-set=utf8mb4 \
  -uroot -proot123 stablepay_payment_db <<'SQL'
SET NAMES utf8mb4;

SELECT tx_id, agent_did, skill_did, tx_hash, status, confirmed_at, updated_at
FROM payment_transactions
WHERE tx_hash = '你的链上tx_hash';
SQL
```
3. 替换语句中 查到的_agent_did、查到的_skill_did、查到的_tx_id 为上一步查询出的真实数据：
```bash
kubectl -n zheda-agent exec -i "$MYSQL_POD" -- mysql \
  --default-character-set=utf8mb4 \
  -uroot -proot123 stablepay_verification_db <<'SQL'
SET NAMES utf8mb4;

INSERT INTO purchase_records (created_at, updated_at, deleted_at, agent_did, skill_did, tx_id)
VALUES (
  NOW(3),
  NOW(3),
  NULL,
  '查到的_agent_did',
  '查到的_skill_did',
  '查到的_tx_id'
)
ON DUPLICATE KEY UPDATE
  tx_id = VALUES(tx_id),
  updated_at = NOW(3);

-- 校验写入结果
SELECT agent_did, skill_did, tx_id
FROM purchase_records
WHERE agent_did = '查到的_agent_did'
  AND skill_did = '查到的_skill_did';
SQL
```
另外我们的codex大人也新增了 **Topic 初始化任务文件**：`infra-deployment/k8s/ack/infra/rocketmq-topic-init-job.yaml`，每次 RocketMQ Broker 重建后执行以下命令初始化 Topic：

# 重新应用初始化任务
kubectl apply -f infra-deployment/k8s/ack/infra/rocketmq-topic-init-job.yaml

### 13.3.4 长期修复方案
#### 用 Job 显式初始化 RocketMQ topic
不要依赖 `autoCreateTopicEnable=true`。
应该把业务 topic 当成基础设施的一部分，在 K8S 里显式声明：
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: stablepay-rocketmq-topic-init
  namespace: zheda-agent
spec:
  template:
    spec:
      restartPolicy: OnFailure
      imagePullSecrets:
        - name: acr-secret
      containers:
        - name: topic-init
          image: stablepay-registry.cn-shanghai.cr.aliyuncs.com/stablepay-dev/rocketmq:5.3.2
          command:
            - sh
            - -c
            - |
              sh /home/rocketmq/rocketmq-5.3.2/bin/mqadmin updateTopic \
                -n stablepay-rocketmq-nameserver:9876 \
                -c DefaultCluster \
                -t payment_events \
                -r 4 -w 4
```
这样 broker 重建后，topic 可以被重新创建。
#### 给 RocketMQ broker 做持久化
RocketMQ broker 不应该长期裸跑在无状态 Deployment 里。
```text
StatefulSet
  +
PVC 挂载 RocketMQ store/config
  +
稳定的 broker identity
```
这样 broker 重启后，topic 配置、commit log、consumer offset 等状态更容易保留。

#### 降低 broker 地址变化对客户端的影响
这次日志里旧 broker IP 是一个强信号。
RocketMQ 客户端会从 NameServer 拉 broker route。  
如果 broker Pod IP 变了，但客户端里还有旧 route，短时间内就会打旧地址。

工程上可以考虑：
- broker 重建后重启 producer/consumer；
- 使用 StatefulSet + 稳定网络身份；
- 在 broker 配置里明确 advertised address；
- 监控 consumer 日志里的 `dial tcp old-ip:10911`。

## 13.4 面试讲法
如果面试官问：“你们支付成功后为什么还会 402？”

我会这样回答：
> 402 在我们系统里不是链上支付状态，而是业务购买态。merchant-backend 会调用 verification-service，只有 purchase_records 里存在 agent_did + skill_did 的记录，才会解锁资源。链上交易成功以后，payment-service 会异步发布 payment_events，verification-service 消费后写 purchase_records。所以如果 MQ topic 缺失、broker 路由异常或 consumer 没消费，即使链上已经转账，verification 仍然 false，商家继续返回 402。

如果继续问：“你怎么定位的？”

我会说：
> 我先确认 merchant 返回 402 是符合逻辑的，然后看 verification 日志。日志里同时出现了 `topic not exist` 和连接旧 broker IP 超时。接着我用 mqadmin 查 NameServer 的 topic route，确认 `payment_events` 不存在；再查 clusterList，发现当前 broker 是新 IP，而 consumer 还在访问旧 IP。所以根因不是业务代码，而是 RocketMQ topic 丢失加客户端路由缓存。现场修复是创建 topic 并重启 payment/verification，长期修复是 topic-init Job、broker 持久化和稳定 broker 地址。

如果问：“你为什么用 MQ？”

我会说：
> 支付确认是异步的，链上最终态和业务购买态不能放在一次同步请求里。MQ 把 payment-service 和 verification-service 解耦，payment-service 只发布支付事实，verification-service 独立消费并落业务购买记录。这样系统是最终一致，但要求我具备处理消息丢失、topic 缺失、consumer lag、broker route 等问题的能力。

如果问：“这次学到了什么？”

我会说：
> Pod Ready 不等于 MQ 可用，端口通不等于 topic route 存在。对于有状态中间件，K8S Deployment 只能保证进程重启，不能保证业务元数据恢复。RocketMQ 这种组件需要 topic 初始化、broker 持久化、route 观测和消费端重启策略，否则很容易出现‘服务全绿但业务链路断了’。
